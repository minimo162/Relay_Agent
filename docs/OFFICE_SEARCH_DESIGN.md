## Goal

Let the agent search the contents of Office files (`.docx`, `.xlsx`, `.pptx`) and `.pdf` as naturally as it currently searches plaintext. No embeddings / no vector DB in this phase — plaintext extraction + regex/substring search only. Semantic search is deferred to a possible future **Phase B**.

## Scope

**In scope**
- Text extraction from `.docx`, `.xlsx`, `.pptx`, `.pdf`
- A shared extraction cache with path-indexed lookup and content-hash invalidation (see **Cache** for the key schema — earlier drafts said "keyed by content hash"; that was revised)
- A new agent tool for searching across Office/PDF files
- Transparent plaintext return from the existing `read_file` tool for Office extensions (matching today's PDF behavior)

**Out of scope (Phase A)**
- Embeddings, vector indexing, semantic ranking
- OCR (consistent with `pdf_liteparse`, which runs OCR-off)
- Legacy binary formats `.doc` / `.xls` / `.ppt` (pure-Rust coverage is poor; may revisit via Windows COM later)
- Images / charts / SmartArt / drawing-canvas text (including DOCX `<w:drawing>` text-box runs, `<mc:AlternateContent>`-fallback glyph runs, XLSX chart titles, PPTX chart text). Phase A extracts only the ordinary body/header/footer/table/notes text stream of each format — anything that lives under a drawing, picture, or chart part is deferred.
- M365 / OneDrive / SharePoint sources (current app has no Graph API integration; Copilot-via-CDP is the existing M365 path and is unrelated to this feature)
- Frontend search palette UI

## Why not "just grep"?

`grep_search` reads raw bytes. `.docx/.xlsx/.pptx` are zipped XML, so `grep` against the file bytes matches compressed noise, not content. The blocker for agentic search over Office files is **parsing**, not ranking. Once text is extracted, iterative LLM querying over BM25/regex already covers most practical retrieval — synonyms/paraphrase cases are the only place semantic search meaningfully wins. We therefore ship plaintext extraction first and re-evaluate before adding embeddings.

## Existing constraints to respect

PDF extraction is already wired. This design aligns with those conventions rather than duplicating them.

| Concern | Existing convention (`pdf_liteparse`) |
|---|---|
| Single entry point | `pdf_liteparse::read_pdf_as_text(path, pages)` |
| Sidecar | `relay-node` + `liteparse-runner/parse.mjs` (@llamaindex/liteparse, OCR off) |
| Node resolution | `RELAY_BUNDLED_NODE` → bundled `relay-node` → PATH `node` |
| Runner root resolution | `RELAY_LITEPARSE_RUNNER_ROOT` → `CARGO_MANIFEST_DIR/../../liteparse-runner` |
| Max output bytes | `MAX_PDF_TEXT_BYTES = 16 MiB` |
| Max stderr bytes | `MAX_PDF_STDERR_BYTES = 32 KiB` |
| Timeout env | `RELAY_PDF_PARSE_TIMEOUT_SECS` (default 120s, max 3600s) |
| Output header | `[PDF text via LiteParse — OCR disabled; spatial/layout-oriented extraction; quality varies by file.]\n\n` |
| `read_file` integration | Extension `.pdf` is dispatched to `read_pdf_as_text` transparently; `pages` param is PDF-only |

Non-negotiable: **PDF extraction remains owned solely by `pdf_liteparse`.** The new Office path must reuse it rather than invoke `parse.mjs` independently.

## Design principles

1. **Single owner per format.** PDF stays with `pdf_liteparse`. `docx/xlsx/pptx` live in a new `office` module. No duplicate codepaths.
2. **Pure-Rust for Office.** Avoid adding Node sidecar round-trips for formats we can parse natively. Faster, more predictable, no extra bundling.
3. **Symmetry with PDF.** `read_file` already handles `.pdf` transparently — it should handle `.docx/.xlsx/.pptx` the same way.
4. **Conventions mirror `pdf_liteparse`.** Byte limits, timeout env var shape, header prefix, error wrapping.
5. **Cache once, use twice.** Both `read_file` and the new search tool read through the same extraction cache.

## Public types

```rust
pub struct ParsedDoc {
    pub source: PathBuf,
    pub format: DocFormat,            // Docx | Xlsx | Pptx | Pdf
    pub anchors: Vec<AnchoredText>,   // granular units for citation
}

pub struct AnchoredText {
    pub anchor: String,               // "p12" | "Sheet1!A1" | "slide3" | "doc" (pdf, phase A)
    pub text: String,
}
```

`office::extract(path)` dispatches:
- `.docx` → `docx::extract`
- `.xlsx` → `xlsx::extract`
- `.pptx` → `pptx::extract`
- `.pdf`  → calls `pdf_liteparse::read_pdf_as_payload(path, None)` and produces a **single document-level anchor** (`"doc"`) whose text is the header-stripped payload. Earlier drafts said `office::extract` called `read_pdf_as_text`, which emits the LiteParse banner header (`apps/desktop/src-tauri/crates/runtime/src/pdf_liteparse.rs:226`). That would have folded the literal header string into every PDF's cache anchor, causing `office_search` regexes like `/LiteParse/` or `/OCR disabled/` to match every indexed PDF. `read_pdf_as_text` remains the `read_file` dispatch arm for `.pdf` (it needs the header for the user-visible view); `office::extract` must use the payload-only path. The Phase A7 follow-up below will lift the document-level anchor restriction; nothing about it changes the header-stripping requirement.

## Extraction details

### docx (`docx.rs`)
- Open as zip, read `word/document.xml`
- **Table-aware traversal (rev 14 sharpening; row-anchor grammar updated in rev 15).** The body-paragraph iterator walks the document-body tree at one level only: a `<w:p>` counts as a body paragraph anchor (`p{1-based index}`) **iff its immediate structural parent is `<w:body>` (or a `<w:sdt>` / `<w:sdtContent>` structured-document-tag wrapper whose parent is `<w:body>`)**. A `<w:p>` whose parent chain passes through `<w:tbl>` / `<w:tr>` / `<w:tc>` is **not** emitted as a body paragraph; it is seen only by the table traversal below. This is the explicit rule that prevents the same cell text from appearing as both a `p{n}` anchor and a `p{n}:tbl{j}:row{i}` anchor — the earlier "iterate `<w:p>` paragraphs" wording did not specify scope and a naive descendant-walk implementation would double-count table text. The paragraph counter advances for body paragraphs only; table rows do not consume paragraph indices.
- Preserve document order across body paragraphs and tables: when a `<w:tbl>` is encountered in the body walk, emit its row anchors inline at that position (see below), then resume the body-paragraph iterator.
- Concatenate `<w:t>` runs within a single anchor's scope. A body paragraph anchor's text is the concatenation of `<w:t>` runs under that `<w:p>` (including runs inside inline field / hyperlink wrappers, excluding runs inside a nested `<w:tbl>` — inline tables are rare but if present, their rows are emitted as their own anchors at the nested position and do not contribute to the surrounding `<w:p>`'s text).
- Include headers/footers (`word/header*.xml`, `word/footer*.xml`) as distinct anchors (`header1:p1`, …). Each header/footer part runs the same table-aware traversal as the body, with its own independent paragraph counter (so `header1:p1`, `header1:p2`, …, `header2:p1`, …). Tables inside headers/footers follow the same "emit row anchors at their position" rule as body tables.
- Skip tracked-changes deletions (`<w:del>`); include insertions (`<w:ins>`). The tracked-changes filter applies to `<w:t>` run content, not to structural decisions — a paragraph that exists only because of an insertion is still emitted.
- **Table traversal.** When the body walk encounters a `<w:tbl>`, iterate its `<w:tr>` children in document order. For each row, the anchor is `p{n}:tbl{j}:row{i}` (rev 15 anchor grammar; see **Anchor uniqueness** bullet below) where `n` is the body-paragraph index *immediately preceding* this table (or `0` if the table is the first body element; `p0:tbl{j}:row{i}` is a legal anchor name under this scheme and is reserved for that case), `j` is the 1-based table index within the part being walked, and `i` is the 1-based row index within the table. The row's text is the concatenation of all `<w:t>` runs under that `<w:tr>`, in document order, across all of its `<w:tc>` cells, separated by the tab character (`\t`) at the cell boundary — the same tab-as-separator convention the xlsx line serialization uses. Nested tables inside a cell are emitted as their own `row` anchors at their nested position (incrementing the part-level `j` counter) and do not contribute to the enclosing row's text.
- **Anchor uniqueness (rev 15 correction).** Row anchors must be unique within a single `ParsedDoc` so that the `office_search` result shape — `{path, anchor, match, preview}` (see **Behavior** step 8) — can round-trip as a citation the caller can act on. Earlier drafts allowed multiple consecutive tables with no intervening body paragraphs to emit repeated anchors like `p1:row1`, and argued "disambiguation is by anchor ordering in the `anchors` vector, which matches document order." That reasoning is wrong for the citation contract: the tool result does not expose the anchor's vector position, only the anchor string and the path, so two matches in consecutive sibling tables under the same preceding body paragraph would produce byte-identical citations. The caller (or user) has no way to distinguish them, and `read_file` cannot resolve a duplicate anchor back to a specific row. Rev 15 therefore requires anchors to be **globally unique within the document**: when the body walk encounters the `j`-th `<w:tbl>` (1-based, counted across all tables in the *part being walked* — body, each header, each footer independently), its row anchors are `p{n}:tbl{j}:row{i}`, where `n` is the body-paragraph index immediately preceding this table (or `0` if this table is the first body element of the part), `j` is the table index within the part, and `i` is the 1-based row index within the table. A lone table still produces `p{n}:tbl1:row{i}`; the `tbl{j}` segment is always present so the anchor grammar is uniform and does not branch on "is this table disambiguated from any other." Nested tables follow the same scheme relative to their enclosing part's table counter — a nested `<w:tbl>` inside a cell increments the part-level `j` counter and emits `p{n}:tbl{j}:row{i}` at its nested position. Headers/footers run their own independent `j` counter alongside their independent paragraph counter, so a header's first table is `header1:tbl1:row{i}` (preserving the "headers/footers are parts" naming convention from the bullet above).

### xlsx (`xlsx.rs`)
- Use `calamine` (read-only)
- For each sheet, emit one anchor per non-empty cell: `{SheetName}!{A1}` → cell value as text
- **Cell values are emitted as-is; formatting is best-effort, not guaranteed.** When `calamine` exposes a pre-formatted string for a cell, we use it; otherwise we use the raw value's `Display` form. We do **not** ship a bespoke number-format engine in Phase A — currency symbols, thousand separators, and locale-specific date styling may therefore differ from Excel's display. This is Option X in the open-questions section below, now promoted from "preferred default" to "the A2 contract." Search recall is the primary concern, not visual fidelity.
- Formula cells: emit the cached value; fall back to the formula string prefixed with `=` when no cached value
- Hard cap: configurable `RELAY_OFFICE_XLSX_MAX_CELLS` (default 500k cells) to bound memory on very large workbooks

### pptx (`pptx.rs`)
- Open as zip, iterate `ppt/slides/slide{N}.xml` in numeric order
- Concatenate all `<a:t>` text runs per slide
- Anchor = `slide{N}`; notes slides → `slide{N}:notes`
- Speaker notes come from `ppt/notesSlides/notesSlide{N}.xml`
- **Malformed notes contract (rev 8 pin; rev 9 env-var unification).** If `notesSlide{N}.xml` fails to parse (XML error, truncated, event-limit breach confined to the notes part) while `slide{N}.xml` parses cleanly, the extractor **omits the `slide{N}:notes` anchor from `ParsedDoc.anchors`** and continues. The slide body anchor is still emitted. No per-anchor error surfaces to the caller: `ParsedDoc` (see **Public types** → line 73) is intentionally anchor-only, and adding an `errors` field to it would bleed into the cache record schema, the `read_file` line-serialization contract, and the `office_search` result shape — all of which currently assume anchors are the sole output. Whole-file failures (body part unparseable, container-level limit breach) still propagate as `InvalidData` from `office::extract`; the omission contract applies *only* when the defect is confined to a non-essential sub-part (currently just notes). The omission is logged via the shared Phase A5 diagnostics sink described under **`office_search` → Operator visibility** — same env-gate (`RELAY_OFFICE_DIAGNOSTICS_LOGGING=1`, default off) as the denylist sink, but on a distinct tracing subtarget `relay::office::diag::pptx_notes` so an operator filtering only one stream sees only that stream. Not surfaced via IPC. If future parsers introduce more optional sub-parts (e.g. chart-text extensions post-Phase-A), they extend this same omission contract and add a new `relay::office::diag::*` subtarget rather than introducing a new per-anchor error channel or a new env var.

### pdf (reuses `pdf_liteparse`)
- Call into `pdf_liteparse` for a **payload-only** view — do **not** duplicate the LiteParse header string on the `office` side. Phase A4 adds:
  - `pub const PDF_LITEPARSE_HEADER: &str = "[PDF text via LiteParse — ...]\n\n";`
  - `pub fn read_pdf_as_payload(path, pages) -> io::Result<String>`
  - `read_pdf_as_text(path, pages)` is rewritten as `format!("{PDF_LITEPARSE_HEADER}{payload}")` over `read_pdf_as_payload`, preserving today's output byte-for-byte.
  - `office::extract` for `.pdf` uses `read_pdf_as_payload`; `read_file`'s existing `.pdf` arm keeps calling `read_pdf_as_text` and therefore still emits the unchanged header. PDF ownership stays with `pdf_liteparse`.
- **Phase A anchor granularity: document-level only.** The extracted payload is stored as a single anchor `doc` containing the entire text. `parse.mjs` today writes `result.text` straight to stdout with no page boundary markers (see `apps/desktop/src-tauri/liteparse-runner/parse.mjs:48`), so any attempt to recover page anchors from the current output would be a heuristic split and would degrade citation quality unpredictably.
- **Asymmetry with Office is intentional and documented.** `office_search` results for `.pdf` will cite `path` only (anchor field = `doc`), while `.docx/.xlsx/.pptx` cite granular anchors. Users needing per-page PDF citations should call `read_file(path, pages=...)` as today.
- **Page-aware PDF anchors is a Phase A follow-up (not A–1 through A–6).** It requires a structured-output API on both sides: extend `parse.mjs` to emit JSONL with `{page, text}` per page (LiteParse exposes `result.pages`), add `pdf_liteparse::read_pdf_as_pages() -> Vec<(u32, String)>`, and switch `office::extract` for `.pdf` to populate one anchor per page. Tracked as Phase A7.

## Cache

### Key design

The cache must support **O(1) lookup from a file path** without hashing the file's contents first. Earlier drafts named cache files by content hash, which made the "cheap `(mtime_sec, mtime_nsec, size)` pre-check" impossible — finding the record required either hashing the file or walking the cache directory. That has been corrected.

- **Cache filename = normalized canonical path hash.** `sha256(normalize_cache_path(canonicalize(path)) as OS-native bytes)` → `{cache_dir}/relay/office_text/by_path/<path_hash>.json`. Given a source path, the corresponding record is read in one `open()` call.
- **Canonical path normalization for the key.** `canonicalize(path)` is not used as a raw display string on Windows. Before hashing or persisting `source`, the cache layer runs the canonical `PathBuf` through a small `normalize_cache_path` helper:
  - On Windows, strip verbatim prefixes returned by `canonicalize`: `\\?\C:\...` becomes `C:\...`, and `\\?\UNC\server\share\...` becomes `\\server\share\...`.
  - Preserve the rest of the canonical path exactly; do not lowercase paths in Phase A. Case normalization is a broader Windows policy question because mixed-case network shares and display casing can be meaningful to users even if lookup is case-insensitive.
  - On non-Windows, return the canonical path unchanged.

  **Path-to-bytes rule (rev 15 clarification).** The canonical path is hashed and stored using **OS-native bytes**, never a `to_string_lossy()` string, so two distinct non-UTF-8 Unix paths cannot collide on the `U+FFFD` replacement-character form of their lossy decodings. Concretely:
  - POSIX: hash `normalized.as_os_str().as_bytes()` (via `std::os::unix::ffi::OsStrExt::as_bytes`). Persist `source` in the record as the base64-encoded form of those bytes paired with `"source_encoding": "unix-bytes-b64"` (see **Record schema** below for the full field layout and the `"utf8"` optimization).
  - Windows: hash `normalized.as_os_str().encode_wide().flat_map(u16::to_le_bytes).collect::<Vec<u8>>()` (via `std::os::windows::ffi::OsStrExt::encode_wide`). Windows `OsStr` is WTF-16, not UTF-8, so `to_string_lossy()` would likewise collide on unpaired surrogates. Persist as base64-encoded wide bytes paired with `"source_encoding": "windows-wide-b64"`.
  - For both platforms, callers that need a display form (diagnostics, error messages) may compute `normalized.display()` lazily from the stored bytes; the cache never relies on a display form for lookup.

  `office_search` glob expansion may produce non-UTF-8 candidates on POSIX (user supplies `paths: ["/data/**"]` and `walkdir` returns entries with arbitrary bytes). These candidates flow into the path-hashing step above without modification; no `to_string_lossy()` step exists. They are **not** separately rejected as `InvalidInput` — there is no reason to fail on "this legitimate file happens to have a non-UTF-8 name" when the downstream cache and `office::extract` both accept `&Path`. The one exception is tool-boundary surfacing: the `office_search` result shape today is JSON and its `path` field is a string, so a non-UTF-8 path in results falls back to `path.to_string_lossy()` purely for the **output** field while the **cache lookup** used the OS-native bytes. This means `office_search` results for non-UTF-8 Unix paths are still caller-readable but may not be round-trippable back through the caller's shell without re-escaping — an accepted Phase A limitation, tracked under open-questions item 32.

  The cache key is `sha256(os_native_bytes(normalize_cache_path(canonicalize(path))))`, and the record's `source` stores the same normalized byte sequence. This avoids splitting one Windows file into two cache records solely because one call path observed a verbatim `\\?\` form and another observed a conventional path, and avoids collapsing two distinct non-UTF-8 Unix files into one cache record.
- **Content hash lives inside the record**, not in the filename. This decouples "which record belongs to this path" from "is the content still the one we extracted."
- `cache_dir` = `dirs::cache_dir()` (platform-standard: `%LOCALAPPDATA%` on Windows, `~/Library/Caches` on macOS, `~/.cache` on Linux)

### Record schema

```json
{
  "source_encoding": "unix-bytes-b64",
  "source": "L2Ficy9wYXRoL3RvL2ZpbGUuZG9jeA==",
  "format": "docx",
  "mtime_sec": 1712345678,
  "mtime_nsec": 123456789,
  "size": 123456,
  "content_hash": "sha256:...",
  "extracted_at": "2026-04-19T12:34:56Z",
  "schema_version": 3,
  "anchors": [{"anchor": "p1", "text": "..."}]
}
```

**`source_encoding` and `source` (rev 15 contract).** The record must preserve the normalized canonical path as OS-native bytes, not as a lossy-decoded string, so that two distinct non-UTF-8 paths cannot alias onto the same `U+FFFD`-collapsed record. `source_encoding` is one of three string tags, and `source` is the corresponding payload:
- `"unix-bytes-b64"` — `source` is the base64-encoded (standard alphabet, padded) form of `normalized.as_os_str().as_bytes()` (POSIX, via `std::os::unix::ffi::OsStrExt::as_bytes`).
- `"windows-wide-b64"` — `source` is the base64-encoded form of `normalized.as_os_str().encode_wide().flat_map(u16::to_le_bytes).collect::<Vec<u8>>()` (Windows, via `std::os::windows::ffi::OsStrExt::encode_wide`).
- `"utf8"` — optional optimization for POSIX paths whose bytes are valid UTF-8 and do not begin with `/dev`, `\\`, or other ASCII sequences that might collide with base64. When chosen, `source` is the path string directly. The loader accepts this tag but the encoder is allowed to always emit `"unix-bytes-b64"` / `"windows-wide-b64"` if the simplicity is preferred; `"utf8"` exists only so human-readable records are possible for the common case.

**Base64 dependency.** `base64` is already a transitive dep in the workspace (via `reqwest`'s feature set and several other crates). A2 will add it as a **direct** `runtime` dep so the cache layer has a stable import path — add `base64 = "0.22"` to `runtime/Cargo.toml`. An alternative considered and rejected is storing `source` as a raw JSON array of `u8` (`[47, 97, 98, ...]`) to avoid the dep; rejected because (a) JSON integer arrays roughly double the stored size versus base64, (b) they make records noisier to eyeball, and (c) base64 is small, MIT-licensed, and widely audited. The record schema's `source_encoding` tag is load-bearing: a reader that sees an unknown encoding must discard the record and re-extract, same as an unknown `schema_version`.

**Rejected alternative: store the display string.** The pre-rev-15 schema had `"source": "/abs/path/to/file.docx"` — a lossy `String`. That form survived as long as "the only paths the tool sees are the ones `canonicalize` returns from user-supplied strings" was true, but `office_search` glob expansion can produce arbitrary-byte file names on POSIX, and lossy decoding collapses distinct entries onto the same replacement-character key. The display form is still useful for diagnostics and for the `office_search` result JSON's `path` output field (see **Behavior** step 8), but it is not what the cache lookup compares.

`mtime` is persisted as a `(seconds, nanoseconds)` pair — the full platform `SystemTime` precision as reported by `std::fs::Metadata::modified().map(|t| t.duration_since(SystemTime::UNIX_EPOCH))`. Prior drafts used a bare `mtime` integer at second granularity, which is unsafe: on platforms where filesystem timestamps have higher resolution than whole seconds (APFS / NTFS / most modern ext4 configurations), a cloud-sync client or editor can rewrite a file with the same byte count within the same second, and a second-granularity cache record would return stale extracted text without ever reaching the hash recheck. Recording both `sec` and `nsec` closes that gap; equality in the cheap pre-check requires **both** fields to match exactly. If only `sec` matches (i.e. the platform happens to report second-granularity or the subsecond field was zero at extraction time but non-zero now), the record falls through to the hash recheck, which handles the rest.

**`schema_version` rev-15 rationale.** Phase A ships at `3`. Rev 5 had previously split `mtime` into `mtime_sec`/`mtime_nsec` and bumped `2`. Rev 15 makes two further record-contract changes: (a) `source_encoding` is added and `source` becomes the corresponding encoded form rather than a display string, and (b) DOCX row anchors change grammar from `p{n}:row{i}` to `p{n}:tbl{j}:row{i}`. Either change alone would require invalidating earlier records. Since no `schema_version: 2` record has ever shipped to end users — Phase A is pre-implementation, so only internal test caches have been written — rev 15 folds all pre-release record-format work into schema `3` rather than incrementing to `4`. The loader discards records whose version it does not recognize, so a developer who had a stray on-disk `schema_version: 2` from an interim build will silently re-extract on first access. If any `2`-era records were to reach a shipping build before A1 lands, the rule becomes strict: bump to `4` and do not retroactively fold. The rev-15 rule ("fold while pre-shipping; bump strictly after first ship") is also the rule that applies to future pre-ship record-format tweaks.

### Lookup / invalidation flow

1. Compute `path_hash` from the OS-native bytes of the normalized canonical absolute path (cheap, deterministic) — see **Key design → Path-to-bytes rule**.
2. Open `by_path/<path_hash>.json`. Missing → extract fresh, write cache, return.
3. **Byte-exact `source` comparison (rev 15).** Decode the record's `source` according to its `source_encoding` tag (base64 → `Vec<u8>` for the `*-b64` tags; take the bytes directly for `"utf8"`). Compare that byte sequence to the OS-native bytes of the currently normalized canonical path using `==` on `&[u8]`, **not** a string comparison. Mismatch (hash collision across different paths, extremely unlikely but cheap to check) → extract fresh, overwrite. Unknown `source_encoding` → discard the record and extract fresh. Never decode `source` via `String::from_utf8_lossy`; that would reintroduce the collision the rev-15 byte form exists to prevent.
4. **Cheap pre-check:** compare current file's `(mtime_sec, mtime_nsec, size)` to the record's. All three must match for a hit — subsecond divergence is a miss, for the reasons given above. If they match, trust the cached extraction and return.
5. **Hash re-check:** if cheap pre-check fails, compute `sha256` of the file on disk. If it equals the record's `content_hash`, the file was touched but not actually changed (common for cloud-sync tools); update `mtime_sec` / `mtime_nsec` / `size` in the record and reuse extraction.
6. Otherwise, re-extract and overwrite the record.

This makes the `(mtime_sec, mtime_nsec, size)` pre-check genuinely cheap (1 stat call) and the hash path only runs when the cheap check fails. Corrupt or unparseable records are deleted and recomputed. No global eviction in Phase A (per-file overwrite only); add an LRU or size budget if real-world usage calls for it.

### schema_version

Bumping `schema_version` on extractor behavior changes lets us invalidate stale payloads without nuking the cache directory. Phase A ships at `3` per the rev-15 rationale above (pre-shipping folds rev-5's mtime split, rev-15's OS-native `source_encoding` field, and rev-15's DOCX `p{n}:tbl{j}:row{i}` anchor grammar into one initial schema). The loader discards records whose version it does not recognize. Post-ship, any extractor-contract change that would produce a stale anchor shape, a different invalidation field, or a different `source` encoding must bump `schema_version`.

## Stopping, limits, and cancellation

`pdf_liteparse` gets stoppability for free because extraction runs in a child process that can be killed on timeout. Office extraction runs **in-process and synchronous** (tools execute via `tools::execute_tool` → `run_read_file`, which is a sync API at `apps/desktop/src-tauri/crates/runtime/src/file_ops.rs:147`). Wrapping the sync call in `tokio::time::timeout` from the outside would return `TimedOut` to the caller, but the extraction thread would keep burning CPU until the OS process exits. That is unacceptable — a malformed or adversarial Office file could wedge the agent loop indefinitely.

### Defense in depth

Three layers, from strongest to weakest:

1. **Bounded per-format structural limits (primary defense).**
   Extraction is written to be linear in the input and inherently bounded. Reject inputs that blow past these limits as early as possible:

   - **Container-level (zip).** The guarantees here rest on **measured decompressed bytes** plus **metadata-based prefilters**. Earlier revs claimed a streamed compression-ratio check driven by a per-entry compressed-byte counter; rev 6 retracts that claim because it is not implementable on top of `zip::ZipArchive`'s API. See "Why not per-entry compressed-byte measurement?" below.

     1. **Metadata values (prefilter only; attacker-controlled).** `zip::read::ZipFile::size()` and `zip::read::ZipFile::compressed_size()` come straight from the central directory. They are cheap to read, but a malicious `.docx` can declare whatever values it wants. Use them to:
        - Short-circuit *obviously oversized* entries before we open a decompressor (e.g. `size() > RELAY_OFFICE_ZIP_MAX_EXPANDED_BYTES` → refuse outright, no decoder allocated).
        - Compute a **metadata-only** ratio `size() / max(1, compressed_size())` as an early rejection signal — if the entry *self-declares* a ratio above `RELAY_OFFICE_ZIP_MAX_COMPRESSION_RATIO`, refuse outright. This catches lazy bombs whose metadata already tells on them; it cannot catch a bomb whose metadata lies. The authoritative guarantee is the measured cap below, not the ratio.

        Never rely on metadata to prove an entry is safe to read — a small `size()` value means nothing if the deflate stream actually expands to gigabytes. Metadata is advisory; the decompressed-byte cap is the hard stop.
     2. **Measured decompressed bytes (enforced, authoritative).** For every entry we read, the `zip::read::ZipFile<'_>` handle (which implements `Read`) is wrapped in a counting adapter:
        - `BoundedRead<R>`: on each `Read::read` call we increment `decompressed_bytes_produced` by the returned byte count, check it against `RELAY_OFFICE_ZIP_MAX_EXPANDED_BYTES` (cumulative across all entries of this archive, not per-entry), and return `io::Error::new(ErrorKind::InvalidData, …)` on breach. This is a ~20-line `Read` wrapper living in `office::zip_guard`; no crate dependency.

        The cumulative decompressed cap is sufficient on its own to bound how much memory/CPU a malicious `.docx` can force us to consume. If the attacker declares honest metadata, the metadata prefilter catches the entry immediately; if they lie about metadata, the measured cap catches them at the moment their deflate stream crosses 256 MiB (default). There is no scenario in which both defenses are bypassed, because the second defense does not depend on the attacker-controlled values at all.

     3. **Entry-name validation before part lookup.** Office XML part paths are matched only after every zip entry name has passed a strict validation helper. Disk writeout is never performed, so classic zip-slip is not the direct risk; the risk is parser confusion, unexpected aliasing of XML parts, or a malformed name triggering a library edge case. The helper enforces:
        - Entry names must be valid UTF-8 as exposed by the `zip` crate. If the crate exposes a non-UTF-8 raw-name path for the selected version, reject that entry as `InvalidData` rather than lossy-decoding it.
        - Reject names containing NUL, backslash, drive prefixes, leading `/`, empty path segments, `.` segments, or `..` segments.
        - Normalize only by validating separators; do not case-fold.
        - Required Office parts are matched by exact, case-sensitive forward-slash names (`word/document.xml`, `xl/sharedStrings.xml`, `ppt/slides/slide1.xml`, etc.). A differently cased or absolute-looking name is not a match; if a required part is absent under the exact name, extraction fails with `InvalidData`.

        The extractor does not rely on `ZipFile::enclosed_name()` alone because that API answers "safe to materialize on disk," while this parser needs an OPC part-name contract. XML parsing for a validated part is wrapped in `std::panic::catch_unwind` at the format boundary and converted to `InvalidData` if `quick-xml`, `zip`, or a parser bug panics; panics must not cross the tool boundary or poison the agent loop.

     **Why not per-entry compressed-byte measurement?** Earlier revs proposed wrapping the archive's *inner* reader — i.e. `zip::ZipArchive::new(BoundedRead::new(file))` — to count compressed bytes per entry, then computing a streamed ratio `decompressed / compressed`. That does not work: `zip::ZipArchive` shares one inner reader across the entire archive. The same `Read + Seek` handle is used to scan the central directory, seek to local-file headers, skip unused entries, and feed every entry's decoder in turn. Bytes consumed by the inner reader therefore do not attribute to any single entry — a ratio built on that denominator can underflow (central-dir reads before we started counting) or overflow (unrelated seeks/header scans attributed to the current entry), producing both false passes and false failures. The `zip` crate does not expose a per-entry "bytes fed into this decoder" accessor; reaching under the crate to add one would require either a fork or unsafe introspection, both of which are out of scope here. Rather than ship a ratio check that silently produces wrong answers, rev 6 uses metadata-only ratio (as a cheap prefilter) and relies on the measured absolute cap as the enforcement.

     - `RELAY_OFFICE_ZIP_MAX_ENTRIES` (default 5_000) — checked against `ZipArchive::len()` before any entry is read. This is a count, not a byte budget; metadata-safe.
     - `RELAY_OFFICE_ZIP_MAX_EXPANDED_BYTES` (default 256 MiB) — measured, cumulative, enforced. Metadata value checked only as the prefilter short-circuit above.
     - `RELAY_OFFICE_ZIP_MAX_COMPRESSION_RATIO` (default 200) — **metadata-only**. Evaluated once per entry from `size() / max(1, compressed_size())` before the decoder is opened. A breach rejects the entry without decompression; a pass does not prove anything, since the measured cap is what guarantees termination.
   - **XML-level:** `RELAY_OFFICE_XML_MAX_DEPTH` (default 256), `RELAY_OFFICE_XML_MAX_EVENTS` (default 5_000_000). `quick-xml` is streamed; the event counter is incremented in the read loop and aborts extraction with `InvalidData` when exceeded. Entity expansion is disabled on the reader (`quick-xml` default).
   - **docx:** `RELAY_OFFICE_DOCX_MAX_PARAGRAPHS` (default 200_000).
   - **xlsx:** `RELAY_OFFICE_XLSX_MAX_CELLS` (default 500_000), plus an **xlsx preflight over the zip container run before `calamine` is handed the file** — see below.
   - **pptx:** `RELAY_OFFICE_PPTX_MAX_SLIDES` (default 2_000).
   - **Output-size guard (all formats):** `MAX_OFFICE_TEXT_BYTES = 16 MiB` on total anchor text, mirrored from `pdf_liteparse::MAX_PDF_TEXT_BYTES`. **On breach, extraction fails with `io::ErrorKind::InvalidData`, reason `"anchor text exceeds MAX_OFFICE_TEXT_BYTES=16777216"`; no partial `ParsedDoc` is returned, no cache record is written, and no per-anchor truncation flag exists on the record schema.** This matches `pdf_liteparse`'s existing error-on-overflow convention (the PDF path errors rather than truncating). Rev 13 retracts the earlier "emission is short-circuited past the limit" wording: silent truncation inside the extractor produced a `ParsedDoc` with missing anchors, which `office_search` would then report as "no match" for content that existed after the cutoff — an invisible search miss the caller had no way to detect. Erroring surfaces the condition at the tool boundary (`InvalidData` reaches the agent as a parse-failure `errors` entry for `office_search`, or as a read failure for `read_file`), so the caller can take a different path (e.g. ask the user to split the file) instead of acting on incomplete results. Files legitimately over the cap are expected to be rare; the limit is set to mirror PDF and to bound cache-record size, not to be a soft guideline. If field data shows the cap is too tight for common inputs, raise `MAX_OFFICE_TEXT_BYTES` rather than reintroducing silent truncation.

   These limits together make extraction terminate in a predictable, small constant factor of the input size for conformant files, and reject adversarial inputs outright.

   **xlsx preflight (runs over the whole workbook, always; preflight and `calamine` share one bounded in-memory snapshot — rev 15 rewrite).** `calamine` loads `xl/sharedStrings.xml` and the active worksheet XML into memory before iteration yields the first cell. That means the per-cell cap at `RELAY_OFFICE_XLSX_MAX_CELLS` runs **after** potentially large allocations. To close that gap without leaving a TOCTOU window between preflight and parse — including the same-inode mutation window that the rev-13/14 shared-`File` scheme could not defend against on POSIX — the procedure is:

   1. Open the `.xlsx` **once** with an `OpenOptions` configured for the snapshot read (see **Concrete constraint** below for the exact POSIX/Windows configuration), then immediately slurp the whole archive into a single `Vec<u8>` bounded by a hard archive-size cap `RELAY_OFFICE_XLSX_ARCHIVE_MAX_BYTES` (default 256 MiB — see note on relation to the decompressed cap in **env var semantics** below). The read uses `file.take(cap as u64 + 1).read_to_end(&mut buf)?` and rejects with `InvalidData` (reason: `"xlsx archive exceeds RELAY_OFFICE_XLSX_ARCHIVE_MAX_BYTES=<cap>"`) when `buf.len() > cap`, so an adversarial file-size lie in metadata cannot inflate the buffer. Drop the `File` as soon as `read_to_end` returns — no `File` handle escapes this step. Every subsequent step reads from the snapshot buffer.
   2. Construct `ZipArchive::new(std::io::Cursor::new(&snapshot))` and walk its central directory, applying the container-level limits above (max entries, metadata prefilters, **cumulative measured decompressed-byte cap during streaming**, metadata-only compression-ratio gate) to all entries. The `Cursor<&[u8]>` is cheap to clone if needed (`cursor.clone()` just copies the position).
   3. Additionally check size ceilings specific to xlsx structure — **evaluated for every sheet in the workbook, independent of any `sheets` filter the caller supplied**:
      - `RELAY_OFFICE_XLSX_SHARED_STRINGS_MAX_BYTES` (default 64 MiB, uncompressed-as-streamed).
      - `RELAY_OFFICE_XLSX_SHEET_XML_MAX_BYTES` (default 128 MiB, per-sheet, uncompressed-as-streamed).
   4. If preflight passes, feed a fresh `std::io::Cursor::new(&snapshot)` (or `Cursor::new(snapshot)` if the parse stage consumes the buffer) to `calamine::open_workbook_from_rs` — the `Read + Seek`-generic entry point. **`calamine` must not be given a path string**, and **`calamine` must not be given a `File` handle** either; it must be given a cursor over the preflighted snapshot buffer. The snapshot is the one and only byte sequence the parse stage sees.

   Because both the zip-container walk and `calamine` consume cursors over the same immutable `Vec<u8>`, any concurrent writer to the on-disk `.xlsx` — whether replacing the directory entry, `ftruncate`-ing the inode, or overwriting bytes in place — is irrelevant once the snapshot read returns. Preflight and parse see byte-for-byte the same buffer.

   **Why bytes-in-memory instead of a shared `File` handle.** Rev 13 introduced a shared-`File` scheme (preflight opens, validates, rewinds, then hands the same handle to `calamine`) to close the rev-12 "hand the path to `calamine`" TOCTOU, in which a concurrent writer could replace the file between preflight-close and calamine-open and slip an oversized `sharedStrings.xml` past the limits. Rev 14 further required `OpenOptionsExt::share_mode(FILE_SHARE_READ | FILE_SHARE_DELETE)` on Windows because Rust's default `share_mode` (`FILE_SHARE_READ | FILE_SHARE_WRITE | FILE_SHARE_DELETE`, per https://doc.rust-lang.org/std/os/windows/fs/trait.OpenOptionsExt.html) would otherwise let a concurrent writer mutate the same file object under our handle on Windows.

   Rev 15 retracts the shared-`File` scheme entirely because it cannot be made safe on POSIX. A POSIX writer that already holds, or later obtains, write access to the same inode can rewrite the bytes between preflight-completion and calamine's reads — no portable share-mode lockout exists. In the worst case the mutated workbook is valid and its newly-written `sharedStrings.xml` or sheet XML exceeds `RELAY_OFFICE_XLSX_SHARED_STRINGS_MAX_BYTES` / `RELAY_OFFICE_XLSX_SHEET_XML_MAX_BYTES` in bytes the preflight never saw, and `calamine` allocates against the un-preflighted bytes before any Phase A limit runs again. The pre-publish stat re-check fires *after* parse and cannot protect memory/CPU safety during parse. A bounded in-memory snapshot does — the bytes calamine sees are the bytes preflight measured, period.

   The one additional protection the snapshot does **not** subsume is the Windows "someone could mutate the bytes while `read_to_end` is still copying them out" race. That race is closed the same way rev 14 closed the equivalent race on the long-lived handle: require `OpenOptionsExt::share_mode(FILE_SHARE_READ | FILE_SHARE_DELETE)` on the `OpenOptions` used to open the snapshot source. The share-mode requirement now covers the *snapshot read* (typically milliseconds) instead of the entire preflight+parse sequence, but the protection mechanism is the same.

   **Concrete constraint.** In the A2 implementation, the snapshot source must be opened as:
   - POSIX (`#[cfg(unix)]`): `std::fs::OpenOptions::new().read(true).open(path)?` — default share/access semantics are sufficient. The immutability of the `Vec<u8>` returned by `read_to_end` is what defends against concurrent mutation, not the open mode.
   - Windows (`#[cfg(windows)]`): `std::fs::OpenOptions::new().read(true).share_mode(FILE_SHARE_READ | FILE_SHARE_DELETE).open(path)?` using `std::os::windows::fs::OpenOptionsExt::share_mode`. Forbidding `FILE_SHARE_WRITE` is the part that matters; keeping `FILE_SHARE_DELETE` lets a concurrent rename/delete of the directory entry still succeed. `FILE_SHARE_READ` = `0x00000001`, `FILE_SHARE_DELETE` = `0x00000004`; the A2 implementation either pulls these from `windows-sys` (`Win32_Storage_FileSystem` feature; see **To add to `runtime`** for the dep bump) or defines them as local `const u32` in `office::xlsx` — both are acceptable, since `share_mode` is a `u32` bitmask and the values are stable Windows ABI.

   After step 1 completes, both preflight and calamine consume `Cursor<&[u8]>` / `Cursor<Vec<u8>>` over the same buffer; **no `File` handle escapes step 1**, and every downstream step is immune to any on-disk activity against `path`.

   The **pre-publish sanity re-check** at **Cache write safety → Pre-publish sanity re-check** protects cache correctness *after* extraction; the bounded snapshot here protects memory and CPU safety *during* parse, which the re-check cannot do because it fires only once the allocations have already happened. Rev 15 makes the "during parse" safety guarantee platform-independent — it rests on buffer immutability, not on a share-mode trick that only works on one OS.

   **Implementation constraint on the zip reader.** `zip::ZipArchive::new` requires `Read + Seek`. With the rev-15 snapshot, preflight constructs `ZipArchive::new(Cursor::new(&snapshot))` for the central-directory walk and limit enforcement, then for the parse stage feeds a fresh `Cursor::new(&snapshot)` to `calamine::open_workbook_from_rs`. Both cursors reference the same immutable byte buffer. **No codepath is allowed to produce `calamine::open_workbook::<_, _>(path)` from a raw `path: &str`** — the existence of that call in the `office` module is a review-rejection criterion. **Nor is any codepath allowed to hand calamine a `File` handle** — the rev-13/14 shared-`File` path is retracted on both platforms and its continued presence in `office::xlsx` is also a review-rejection criterion.

   This pushes rejection of pathological workbooks *before* `calamine` allocates, not after, and removes the window in which a replaced-under-us file could slip past the limits. The alternative — "trust `calamine` + cover it with adversarial fixtures" — is explicitly rejected here; the adversarial fixtures at each parser phase (A1/A2/A3 — see Rollout) remain valuable but are a second line of defense, not a substitute for the preflight.

   **Why filter-independent.** Rev 6 resolves an earlier inconsistency. Extraction and cache population are always **whole-workbook**: `office::extract(path)` produces anchors for every non-empty cell of every sheet, regardless of any caller-supplied `sheets` list, and the cache record always holds that full anchor set. `sheets` is a **read-time filter only**, applied when materializing the line stream for `read_file` or when choosing which anchors `office_search` evaluates. That choice forces preflight to apply to every sheet — if preflight only covered filter-surviving sheets, the cache would fill with anchors for sheets whose XML may or may not have been safe to parse, and the first subsequent call with a different (or no) `sheets` value would replay the parse against those un-preflighted sheets at cache-hit time. Making the cache "whole-workbook" and preflight "whole-workbook" together is the only self-consistent option. The alternative (filtered extraction paths with a cache-key dimension for the filter) is rejected in Phase A: it would require two parse passes for the common case of "read one sheet, later search all of them" and double the security surface.

2. **Inline deadline checks at natural boundaries.**
   Each extractor threads a `Deadline` (absolute `Instant`) through its inner loops and checks it at cheap granularity: per paragraph in docx, per sheet row batch in xlsx (every N rows, e.g. 1_000), per slide in pptx. On expiry: abort with `io::ErrorKind::TimedOut`. This gives cooperative cancellation without relying on thread interruption.

3. **External wall-clock enforcement (last resort).**
   `office::extract_with_timeout(path, deadline)` runs the synchronous extractor on a dedicated `std::thread::spawn`, delivering the result through an `mpsc` channel. The caller waits on `recv_timeout(duration)`. If the timer fires before the thread completes:
   - Return `io::ErrorKind::TimedOut` immediately — the agent loop is not blocked.
   - The worker thread is **detached** (not killed; Rust cannot safely interrupt a thread mid-operation). It will terminate on its own at the next inline deadline check, which by construction happens within a bounded number of events. Layer 1's hard structural limits guarantee this is a finite wait.
   - **The detached worker does not publish to the cache.** Cache publication is gated on (a) the extractor returning `Ok` within the deadline and (b) a pre-publish re-check that the `(mtime_sec, mtime_nsec, size)` observed at extraction start still matches the file on disk. If either fails, the extracted payload is dropped.
   - **In-flight reclaim slot.** The per-path entry in the in-flight table (see **Cache write safety** below) is **not released when `recv_timeout` returns**; it is released only when the worker thread actually exits (via a `JoinHandle` parked on a dedicated reclaim thread, or a one-shot "worker done" channel the next caller polls). This is the mitigation for the otherwise-fatal failure mode: if the lock were released on timeout, a caller could `office::extract(same_path)` again immediately, spawn a **second** worker against a file whose first worker has not yet reached an inline-deadline checkpoint, and so on. For a pathological input that takes close to the structural-limit budget to notice (~hundreds of ms to seconds even with limits in place), repeated timeout-then-retry by the agent loop would multiply CPU and memory headroom linearly with retry count — turning the layer-3 guard from a DoS defense into a DoS amplifier. Keeping the slot occupied until the worker exits makes retries cost at most one outstanding worker per path.
   - **Subsequent calls observe the reclaim.** While the worker is still alive post-timeout, the next `office::extract(same_path)` returns `io::ErrorKind::ResourceBusy` (or `TimedOut` with a distinct message, e.g. `"still cancelling previous extraction of <path>"`) **without** spawning a new worker and **without** blocking the caller. The agent sees an actionable error, not a hang and not a silent fan-out. Only once the prior worker publishes to the reclaim channel does the slot free up and extraction resume normally.
   - **Per-path reclaim timeout.** If a worker fails to reach an inline-deadline checkpoint within `RELAY_OFFICE_RECLAIM_TIMEOUT_SECS` (default `2 * RELAY_OFFICE_PARSE_TIMEOUT_SECS`, hard cap 3600), the reclaim thread logs a diagnostic (path, elapsed, last observed event count) and **continues waiting**. This bounds wait by the layer-1 structural limits, which are themselves bounded. We do not escalate to process-level kill in Phase A — "one stuck worker per pathological path" is the acceptable worst case because layer 1 caps the damage.
   - **Rate-independent of caller retries.** Because the slot is reclaim-gated, an agent loop spamming `office::extract` against a timed-out path cannot inflate the worker count. Memory usage is O(one ongoing extraction per distinct path currently timing out), not O(retries).

### Cache write safety

Two concerns: (a) a detached post-timeout worker must not be able to publish stale data, and (b) two callers racing on the same path must not corrupt each other's writes. Design:

- **No write-through from the detached path.** The worker in layer 3 writes only to an in-memory `ParsedDoc` that it sends on the channel. The **caller**, not the worker, performs cache publication after it receives the payload from `recv_timeout`. If `recv_timeout` returns `RecvTimeoutError::Timeout`, publication is skipped entirely. The worker has no handle to the cache file.
- **Per-path in-flight guard (reclaim-aware).** A process-wide `Mutex<HashMap<PathHash, InFlightSlot>>` holds one slot per canonical path. `InFlightSlot` carries both the extraction mutex and a `reclaim: Arc<Once<ExtractionOutcome>>`-style gate that is signalled **only by the worker thread on actual exit**, not by the caller on `recv_timeout`. Behavior:
  - **Normal case (no timeout).** Caller acquires the slot, spawns the worker, receives the result, publishes cache, signals reclaim, and drops the slot. Next caller proceeds as usual.
  - **Timeout case.** Caller's `recv_timeout` fires, caller returns `TimedOut` to the agent, **but does not signal reclaim and does not drop the slot**. The slot stays occupied until the worker's own inline deadline check (or structural-limit breach) causes it to exit and signal reclaim. A dedicated reclaim thread parks on the slot's `JoinHandle` so the main allocation path never blocks on cleanup.
  - **Second concurrent call while worker still alive.** Observes an occupied slot with `reclaim` not yet signalled → returns `ResourceBusy` immediately. Does not spawn a second worker. Does not block.
  - **Post-reclaim call.** Observes the slot freed (reclaim signalled) or the slot removed → proceeds via the normal lookup flow. The previous worker's payload is **always dropped** on the timeout path, per the layer-3 rule above — a post-timeout call that finds the slot free will re-extract from scratch. This is deliberate: letting a late worker publish would reintroduce the "detached worker writes cache" hazard we explicitly disallowed.
- **Atomic file publish.** Cache writes go through a temp-file-and-rename dance:
  1. Write the JSON to `by_path/<path_hash>.json.tmp-<pid>-<nonce>` in the same directory.
  2. `fsync` the temp file.
  3. Atomic `rename` to `by_path/<path_hash>.json` (atomic on POSIX and on Windows via `MoveFileEx` with `REPLACE_EXISTING` — `std::fs::rename` uses this on Windows).
  4. On any error in steps 1–3, delete the temp file; never leave a partial record at the final name.

  Readers see either the previous complete record or the new complete record, never a torn write.
- **Pre-publish sanity re-check.** Immediately before rename, re-stat the source file. If `(mtime_sec, mtime_nsec, size)` diverges from what extraction observed, discard and re-extract — the file changed mid-extraction and the new payload is stale by definition.
- **Generation token (belt-and-suspenders).** Each `extract_with_timeout` call produces a monotonic `generation` value (atomic `u64`) that is attached to the payload. The cache-publish path inside the caller only proceeds if the current call's generation matches the one still "in flight" for that path; an older generation returning late (shouldn't happen given the channel model, but cheap to check) is dropped.

### Process-wide extraction concurrency cap

The per-path reclaim slot above prevents *per-path* retry fan-out — repeated calls against the same pathological file cost one worker, not N. It does **not** bound the total number of distinct-path workers alive at once. A caller walking a directory tree containing 200 pathological `.docx` files could legitimately kick off 200 concurrent extractions, each carrying its own structural limits worth of CPU/memory headroom; layer 1 caps the damage per worker but multiplies across workers.

Phase A1 adds a process-wide counting semaphore (`tokio::sync::Semaphore`, acquired from sync code via `try_acquire_owned()` only — never `acquire_owned().await` or a blocking park, because the acquire contract is fail-fast per the Acquire semantics bullet below):

- **Dependency prerequisite (rev 7 correction).** `tokio::sync::Semaphore` requires the `sync` Tokio feature, which is **not** currently enabled in `apps/desktop/src-tauri/crates/runtime/Cargo.toml:14` (the crate compiles today against `["io-util", "macros", "process", "rt", "rt-multi-thread", "time"]`). `num_cpus::get()` is also not in the crate's dependency list. A1 therefore must, as part of the same PR that introduces the semaphore:
  1. Add `sync` to the `tokio` feature list in `runtime/Cargo.toml`.
  2. Add `num_cpus = "1"` as a direct `runtime` dependency.
  These two additions are tracked under **Dependencies → To add to `runtime`** below. Shipping the cap code without them will break the runtime crate build.
- `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS` (default `min(4, num_cpus::get().saturating_sub(1).max(1))`, hard cap 32). Tuned conservatively because Office extraction is CPU-bound and each worker can hold tens of MiB of XML state.
- **Acquire semantics (fail-fast, no queue).** Every entry to `office::extract_with_timeout` calls `Semaphore::try_acquire_owned()`. If no permits are available, the caller **immediately** returns `io::ErrorKind::ResourceBusy` with reason `"extraction concurrency cap reached; retry later"` — there is no internal wait, no bounded park, and no queue. This shape matches the "slot busy" return from the per-path reclaim check, so callers handle both uniformly. Fail-fast is the deliberate choice: queuing N agent-loop callers behind a 4-permit semaphore would trade the agent loop's responsiveness for throughput we cannot guarantee the agent actually wants — the agent may have already changed its mind about the `office_search` call by the time a permit frees. Retry scheduling, if any, belongs on the caller side, not inside the runtime.
- **Release semantics.** The permit is released on extractor exit (success **or** failure **or** timeout-reclaim). Specifically, if the layer-3 timeout fires, the permit is held until the detached worker actually finishes — the semaphore tracks real in-flight workers, not caller intent. This is intentional: a caller that timed out and a caller whose worker is alive both cost CPU, so both should count against the cap.
- **User-visible consequence after a timeout.** If a tool-wide wall-clock cap fires or a per-file timeout returns while extractor workers are still reclaiming, those workers keep holding their semaphore permits until they actually exit. During that reclaim window, later Office reads or searches can return `ResourceBusy` for several minutes even though the previous user-visible call has already returned. This is expected back-pressure, not a deadlock: the permits represent real in-flight CPU/memory work. Release notes must call out this behavior so operators understand transient `ResourceBusy` spikes after a timed-out `office_search`.
- **No fairness ordering (rev 8 retraction).** Earlier revs called out "FIFO fairness so a long queue of callers cannot starve." That was wrong given the fail-fast acquire: there is no queue for FIFO to apply to. Under contention, which caller wins an in-flight permit as it releases depends on (a) whose `try_acquire_owned()` reaches the atomic first and (b) whatever scheduling order `tokio` applied to the calling tasks. Callers must treat `ResourceBusy` as the *normal* outcome under contention, not as a rare error. Starvation in the "one caller never gets a permit" sense is bounded structurally: every permit-holder terminates within `RELAY_OFFICE_PARSE_TIMEOUT_SECS` by the layer-1/2/3 machinery, so any retrying caller acquires *within a bounded window* in expectation, even if not in any particular order.
- **Relationship to the per-path slot.** The semaphore cap is orthogonal to the per-path in-flight guard. A permit is acquired *after* the per-path slot has been reserved — if the per-path slot is already occupied (active worker or mid-reclaim), the caller returns `ResourceBusy` without touching the semaphore. This avoids permit waste on retries that were going to fail per-path anyway.
- **Per-path slot release on permit-acquisition failure (rev 7 correction; must-have for correctness).** Because the per-path slot is reserved *before* the global permit is acquired, a cap miss at permit-acquisition time leaves a reserved slot with no worker behind it. Literal reading of earlier revs would let that slot stay occupied forever (the reclaim channel is only signalled by an exiting worker, and no worker was ever spawned), so the next `office::extract(same_path)` would see `ResourceBusy` perpetually. The required rule: **on any failure to acquire the global permit, the caller must remove the reserved per-path slot from the in-flight table (or explicitly mark it as "no worker spawned, reclaim complete") before returning `ResourceBusy`.** Equivalently, the implementation can defer insertion of the per-path slot until *after* the global permit is successfully acquired; either ordering is acceptable as long as a permit-miss does not leave a ghost slot behind. The slot-retained-until-worker-exits rule from **Cache write safety → Per-path in-flight guard** applies only when a worker was actually spawned.

### Anchor → line serialization (contract for `read_*_as_text`)

`read_file` is a line-oriented API — `offset` and `limit` count lines. Office anchors are structural (paragraph, cell, slide), so `read_*_as_text` has to define a **stable, snapshot-friendly line serialization** that downstream `offset` / `limit` arithmetic, compat-harness goldens, and prompt examples can depend on. This contract is frozen with each phase's schema bump; extractor changes that would alter the serialized line layout require a `schema_version` bump and new goldens.

Grammar and invariants (applies to all three formats):

- One anchor → one or more **lines** terminated by `\n` in extractor output. The per-format header (prepended by `read_file`, not by the extractor) counts as its own lines, matching today's PDF behavior.
- The anchor identifier is never a separate line. It appears only as a prefix on the value line in xlsx, or as a section marker in docx/pptx (see per-format rules). This keeps `offset`/`limit` math independent of anchor identifier length.
- **Empty anchors are omitted entirely** — no blank "placeholder" line. This keeps adversarial "millions of empty cells" inputs from inflating output line count; it also matches `grep_search`-style intuition that empty content is not a hit.
- **Tab (`\t`) is the in-line separator; LF (`\n`) is the record separator.** Values containing embedded tabs or line breaks are emitted with those characters replaced by a single space; the original value is still searchable through `office_search` because `office_search` operates on `anchor.text` from the cache, not on the line-serialized form. This replacement is documented in the per-format header to avoid user confusion.
- **Section markers** (see below) are fixed ASCII strings, unique enough to be safe anchors for tooling. They occupy their own line.

Per-format rules:

- **docx (`read_docx_as_text`)**
  - One paragraph anchor (`p{n}`, `p{n}:tbl{j}:row{i}`, `header1:p{n}`, `header1:tbl{j}:row{i}`, `footer1:p{n}`, `footer1:tbl{j}:row{i}`, …) → one line of its concatenated text, in document order: body first, then each header part in header-number order, then each footer part in footer-number order, then each table row (emitted inline at the table's position in body order). The `tbl{j}` segment makes row anchors globally unique within the part (see **Extraction details → docx → Anchor uniqueness**).
  - A section marker precedes each non-body region: `--- header1 ---`, `--- footer1 ---`. Body has no leading marker. Tables have no marker; their row anchors interleave with surrounding paragraphs in document order.
  - `anchor.text` in the cache is identical to the line content (no marker prefix). The anchor identifier is reconstructable from the marker + running paragraph index; `office_search` reports it directly rather than making callers parse lines.
- **xlsx (`read_xlsx_as_text`)**
  - One cell anchor → one line of the form `{SheetName}!{A1}\t{value}`. Sheets emitted in workbook order; within a sheet, cells emitted in row-major order over the used range (empty cells skipped per the invariant above).
  - Sheets are separated by a `--- sheet: {SheetName} ---` marker line that precedes the first cell line of each sheet (even if `sheets` is `None`). This gives `offset`/`limit` callers a deterministic anchor point per sheet without having to parse `{SheetName}!A1` prefixes.
  - `anchor.text` in the cache is the cell **value only** (no `Sheet!A1\t` prefix); `office_search` reports the anchor separately. This keeps the cache text identical to what a regex engine should see as "cell content."
- **pptx (`read_pptx_as_text`)**
  - One slide body anchor (`slide{N}`) → one or more lines (one per text-run paragraph), in slide order.
  - One slide notes anchor (`slide{N}:notes`) → same rule, emitted immediately after the slide body, under its own marker.
  - Markers: `--- slide {N} ---` before the body, `--- slide {N} notes ---` before the notes. Missing notes part means the notes marker is not emitted.

Relationship to `offset` / `limit`:

- `offset` / `limit` apply to the **serialized line stream described above**, not to anchors. A compat-harness snapshot that freezes a `read_file` call with `(offset, limit)` is by construction a snapshot of the line serialization; changing the serialization requires a schema bump.
- `office_search` bypasses this layer entirely (it reads from `anchor.text` via the cache), so xlsx searches hit cell values cleanly without the `Sheet!A1\t` prefix contaminating regex matches.

`sheets` and `slides` filtering semantics:

- **`sheets` (xlsx only).** Format: comma-separated list of sheet names, matched case-sensitively against the workbook's sheet names. Unknown names are an `InvalidInput` error that names the unknown entry (so the model can self-correct), not a silent drop. When `sheets` is supplied, the line stream contains **only** the `--- sheet: {SheetName} ---` markers and cell lines for the named sheets, in the order the sheets appear in the workbook — not the order the caller listed them, so the serialization stays stable regardless of how the caller permutes the list. **The filter affects only the read stream — `RELAY_OFFICE_XLSX_SHEET_XML_MAX_BYTES` (and the rest of the xlsx preflight / container-level limits) is always evaluated for every sheet in the workbook before `calamine` sees any cell, independent of `sheets`**. Earlier revs ambiguously said preflight ran "for the sheets that survive the filter"; that wording is retracted in rev 7 because it would let a caller-supplied filter change whether a workbook is parsed/cached, contradicting the whole-workbook cache model in **Stopping, limits, and cancellation → xlsx preflight → Why filter-independent** and the A2 filter-independence fixture at the bottom of this document. If any sheet's XML breaches the preflight limit, extraction fails with `InvalidData` regardless of whether that sheet is in the `sheets` list.
- **`slides` (pptx only).** Format: comma-separated list of 1-based slide indices, or ranges like `3-5` (inclusive). Out-of-range indices are `InvalidInput`. Filter semantics match `sheets`: only the matching `--- slide {N} ---` / `--- slide {N} notes ---` markers and their content appear in the line stream, in ascending slide-number order. Notes are included for a selected slide iff the underlying notes part exists; specifying a slide does not materialize an empty notes section.
- **No `paragraphs` filter for docx in Phase A.** If a caller wants a paragraph range, `read_file(path, offset, limit)` over the serialized stream is the supported path — matching the way `read_file` behaves for plaintext files today. Introducing a docx-specific filter before a user need justifies it would duplicate the `offset` / `limit` contract.
- `sheets` / `slides` affect **both** the serialized stream returned by `read_file` **and** the set of anchors that `office_search` will evaluate when called with the same filter (see the corresponding `office_search` input if/when it ships; Phase A5 ships without a filter and searches all anchors). **Extraction, preflight, and cache population are always whole-workbook / whole-deck** — filters are applied on read, not on cache fill. See **Stopping, limits, and cancellation → xlsx preflight → Why filter-independent** for the security rationale; the short version is that a filtered cache would either replay parse at hit time against un-preflighted content, or require a per-filter cache-key dimension that doubles the security surface.

The non-UTF-8 error message is updated **incrementally per phase** to name only the extensions that are already wired. After A3 it reads:
> `file is not valid UTF-8; try a supported type (.pdf, .docx, .xlsx, .pptx, .ipynb, png/jpg/gif/webp/bmp) or convert to text first`

## New tool: `office_search`

Targets `.docx / .xlsx / .pptx / .pdf`. Plaintext `grep_search` remains the tool of choice for everything else.

**Inputs**
- `pattern` (string, regex)
- `paths` (array of paths or globs). **Accepts both absolute paths and workspace-relative paths**, matching `grep_search` / `glob_search` conventions (`apps/desktop/src-tauri/crates/runtime/src/file_ops.rs:460, 503`, which resolve relative input against the current working directory). Absolute paths are used verbatim; relative paths are joined to the current working directory before canonicalization. Documenting this explicitly so callers don't guess the shape.
- `case_insensitive` (bool, default false)
- `max_results` (int, default 50, hard cap 500) — caps total match rows returned.
- `max_files` (int, default 50, hard cap 1_000) — caps how many candidate files will be extracted and searched in a single call. Distinct from `max_results` because a single file can contribute many matches. If the post-denylist candidate set exceeds `max_files`, the result has `files_truncated: true` (see Behavior step 4.4; rev 11 moved truncation to after the denylist so sensitive-file existence can't flip the flag). **Rev 9 default correction.** Rev 8 set this to 500 with a hard cap of 5_000. With the concurrency cap of 4 and the 120s per-file timeout, the rev 8 worst case was `ceil(500 / 4) * 120s = 15_000s ≈ 4h10m` per call — *hours*, not the "minutes" the doc claimed. Rev 9 drops the default to 50 (worst case `ceil(50 / 4) * 120s = 1_560s ≈ 26m`) and the hard cap to 1_000 (worst case ≈8h — still long, which is why the tool-wide wall-clock cap below is the real guard). Callers who need a larger run should either raise the cap deliberately and accept the wait, or batch `paths` on the agent side.
- `context_chars` (int per side, default 80, hard cap 2_000)
- `include_ext` (array, default `["docx","xlsx","pptx","pdf"]`) — validated against the supported set at step 1 below.

**Behavior**
1. **Validate inputs (no I/O yet).** Before any filesystem call:
   - **Compile `pattern` (rev 11 addition).** Build the `regex::Regex` (honoring `case_insensitive`) at step 1, not at step 7. An invalid regex must fail as `InvalidInput` with the compiler's error message here, before any glob expansion, canonicalization, or extraction runs. Rev 10 left regex compilation implicit at the search step, so a malformed `pattern` would have paid the full glob-expansion + denylist + extraction cost before failing — wasteful at best and at worst a side-channel (the time to reach the regex error depends on how many files were touched). Compile once; reuse the compiled `Regex` at step 7.
   - Validate / normalize `include_ext` — lowercase, strip any leading `.`, reject entries outside the supported set `{docx, xlsx, pptx, pdf}` with `InvalidInput`.
   - Validate `paths`: reject empty, reject entries that are neither a glob (contains `*`, `?`, `[`) nor a concrete path, reject more than 100 entries per call with `InvalidInput`.
   - Validate numeric caps against their hard bounds (`max_results ≤ 500`, `max_files ≤ 1_000`, `context_chars ≤ 2_000`). **Rev 10 correction:** rev 9 lowered `max_files` to default 50 / hard cap 1_000 at the Inputs section but left the old `≤ 5_000` cap at this validation site. Both now agree on `≤ 1_000`.
2. **Pattern-level extension prefilter.** For each `paths` entry that is a **concrete path** (no glob metacharacters), if the file extension is not in the validated `include_ext` list, drop the entry before any filesystem call. This is the only step that can cheaply skip I/O; glob entries cannot be filtered at pattern level because the extension is not known until expansion. (Resolves the rev-6 open question about where `include_ext` sits in the pipeline; rev 8 tightens the wording — only *concrete* paths skip filesystem work here, not globs.)
3. **Expand globs (this is already filesystem I/O).** Glob expansion walks directories via `walkdir` / `glob`, which necessarily stats every traversed entry. To bound the blast radius of an overbroad glob like `paths: ["/**/*"]`:
   - **Pre-expansion pattern denylist (rev 9 addition; rev 10 spec rewrite).** Before `walkdir` / `glob` touches the filesystem, evaluate each glob pattern's **filename component** (the path segment after the last `/`, which is what will be matched against each candidate's basename at expansion time) against an over-approximation of `reject_sensitive_file_path`'s basename rules. Rev 9 spec'd "the longest prefix containing no glob metacharacters," which was unimplementable for the attack we were trying to prevent: `/etc/.env*` has literal prefix `/etc/` — the sensitive basename `.env*` lives in the glob portion, so a prefix-only check would never see it. Rev 10 replaces the spec with a filename-segment check that *can* see `.env*`.

     **Procedure.**
     1. Normalize path separators: split each `paths` entry on **both `/` and `\`** before taking the last segment as `filename_glob`. This is load-bearing on Windows: a glob like `C:\Users\x\.env*` has no `/`, so a POSIX-only split would return the whole string as `filename_glob`, miss the `.env*` tail, and let the pattern enter expansion. Rev 10 spec'd `/`-only split, which would have reopened this side channel on Windows. (For a concrete path with no metacharacters, step 2 of the Behavior already handled extension filtering; this rule applies only to glob entries.)
     2. Lowercase `filename_glob` before comparison. `reject_sensitive_file_path` at `apps/desktop/src-tauri/crates/runtime/src/tool_hard_denylist.rs:360` uses `to_ascii_lowercase()` on the basename; the pre-expansion check must mirror that, otherwise `paths: ["/etc/.ENV*"]` or `paths: ["/tmp/ID_RSA*"]` would skip the pre-expansion denylist and reach the expansion-cap oracle path.
     3. Over-approximation match against the basename denylist, using the lowercased `filename_glob`:
        - starts with `.env` → drop (matches the `.env*` rule).
        - starts with `id_rsa` → drop (matches the `id_rsa*` rule).
        - ends with `.key` or `.pem` → drop (matches the `*.key` / `*.pem` extension rules).
        - equals `*` or `**` (i.e. matches any name) **and** the pattern would be expanded recursively — do **not** drop at this step; expansion's per-candidate denylist will filter individual basenames. Over-blocking every `*` glob would make the tool unusable.
     4. Dropped patterns are **silently removed from the candidate set** — no `InvalidInput`, no `errors` entry, no distinguishing field, exactly like the per-candidate silent-drop in step 5.

     **Why this is an over-approximation, and why that's the correct tradeoff.** A pattern like `/tmp/.enveloper*.docx` contains the literal tail `.enveloper*.docx` which does *not* match the real basename rule (the rule is `basename.starts_with(".env")` → true for `.enveloper*`). That's a false positive: the glob is legitimate but gets silent-dropped. We accept the over-block because:
     - The basename denylist is the authoritative rule; the pre-expansion check mirrors its starts-with / ends-with shape.
     - A caller who genuinely needs `.enveloper*.docx` can request it as a concrete path (which skips this check) or through a parent glob that excludes the basename prefix.
     - Under-approximating here reopens the side-channel oracle we are trying to close.

     **Side-channel rationale (unchanged from rev 9).** Without this pre-expansion check, `paths: ["/somewhere/.env*"]` would enter expansion. If it matched N > `RELAY_OFFICE_MAX_GLOB_EXPANSION` entries, the next step's `InvalidInput` reason-string would name the pattern and become an existence/count oracle. Shutting sensitive-basename globs down before expansion closes that side channel. Patterns whose filename portion does not match the denylist (e.g. `/home/x/.ssh/**/*.pdf`) *do* proceed to expansion — this is consistent with the basename-only denylist limitation called out under **Sensitive-path denylist scope** in the Open questions; `.pdf` inside `.ssh` is not blocked today, and the pre-expansion check deliberately does not paper over that gap.
   - **Expansion cap.** Hard-stop enumeration at `RELAY_OFFICE_MAX_GLOB_EXPANSION` entries per call (default 50_000, hard cap 250_000). Breach → abort expansion, return `InvalidInput` with reason `"glob expansion exceeded RELAY_OFFICE_MAX_GLOB_EXPANSION; narrow the path pattern"`. This short-circuits "enumerate the entire disk" patterns without silently dropping results. **This path is only reachable after the pre-expansion denylist has already silent-dropped any sensitive-literal glob**, so the `InvalidInput` reason cannot be an oracle for sensitive files — it can only name globs that passed the pre-expansion check.
   - **Extension filter during traversal.** Apply the validated `include_ext` filter as each candidate is yielded, so the expansion cap is spent on matching extensions rather than on irrelevant files. The set of accepted extensions is small and fixed, so this does not materially slow traversal.
   - **No directory-traversal skip list (rev 9 retraction).** Earlier wording said the walker "skips any directory whose name matches the `reject_sensitive_file_path` directory-prefix patterns (e.g. `.ssh`, `.gnupg`)." That was incorrect on two counts: (a) the current `reject_sensitive_file_path` at `apps/desktop/src-tauri/crates/runtime/src/tool_hard_denylist.rs:356` is strictly **basename-only** — it matches `.env*`, `id_rsa*`, extensions `.key` and `.pem`, and has no notion of directory-prefix patterns at all; and (b) inventing a parallel directory denylist inside `office` would leave two sources of truth for "sensitive path" that could drift. Phase A therefore does **not** try to skip sensitive subtrees during traversal. The per-candidate denylist at step 4.1 (plus the post-canon re-check at 4.3) is the only security gate; it is basename-only by design, and a `.pdf` sitting inside `~/.ssh` is currently **not** blocked by `reject_sensitive_file_path` because the basename does not match any pattern. This is a known limitation, tracked under **Sensitive-path denylist scope** in the Open questions section below — extending the denylist to cover directory-prefix patterns is Phase A5-adjacent work if we decide we want it, not a Phase A assumption. Rev 10 switched this reference from the fragile numeric form (which was mis-read as "#14 missing" during a recent review) to the named anchor so the cross-link doesn't break when the open-questions list is renumbered.
   - **Security-boundary reminder (rev 8, still valid in rev 9).** Glob expansion is filesystem work; earlier wording that claimed `include_ext` ran "before any I/O or security check" was incorrect for glob inputs and is retracted. The security invariant `office_search` relies on is that **every expanded candidate flows through the per-candidate denylist at step 4.1 (and the post-canon re-check at 4.3) before any read/parse**, and that sensitive-literal globs are silent-dropped at the pre-expansion check above. Extension filtering is a cost-minimization step that shrinks the candidate set; it is not a security boundary and does not need to be.
4. **Per-candidate gate (applied to every surviving candidate, in this fixed order).**
   1. **Pre-canonicalization denylist check.** Call `reject_sensitive_file_path` against the **non-canonical** candidate path (the path the caller supplied, or the glob-expanded path, joined with the current working directory if relative). This is the same ordering `read_file` uses at `apps/desktop/src-tauri/crates/runtime/src/file_ops.rs:153`: the denylist runs *before* `canonicalize()`. Rejected candidates are dropped silently per step 5 below. The reason this ordering is load-bearing: `canonicalize` fails with `NotFound` for missing files but succeeds for existing ones. If rejection ran only post-canonicalize, a probe like `paths: ["/x/.env"]` would canonicalize successfully on an existing sensitive file (and then be silently rejected) but fail loudly with a canonicalize error on a missing sensitive file — a one-bit existence oracle exactly like the one step 5 is designed to prevent.
   2. **Canonicalize.** If `canonicalize()` itself errors on a candidate that had *already* matched a sensitive-path pattern at step 4.1, the error is swallowed and the candidate is dropped silently (same "empty-result" shape as a successful pre-canon rejection). For a candidate that was **not** flagged sensitive in step 4.1, canonicalize errors go into the parse-failure path (step 5 below → `errors` array) as `NotFound`-style reasons, which is safe because the caller supplied that specific path and already knows it.
   3. **Post-canonicalization denylist re-check.** Run `reject_sensitive_file_path` again against the canonical path. This catches candidates that passed the pre-canon check only because they were framed as a non-sensitive symlink pointing at sensitive storage (e.g. `workspace/logs -> ~/.ssh`). A hit here is also silent per step 5.
   4. **Candidate budget (rev 11: moved to *after* the denylist).** Over the **post-denylist surviving set** only — i.e. the list of candidates that passed steps 4.1 through 4.3 — if the count exceeds `max_files` (input; **default 50, hard cap 1_000** — see Inputs above), sort by `(path lexicographic order)` for determinism, keep the first `max_files`, and flag `files_truncated: true` on the result. **The truncation flag must be computed on the post-denylist set, never on the pre-denylist set.** Rev 10 ran truncation before the denylist so the tool "didn't waste denylist evaluations on entries it would drop" — that ordering was a 1-bit existence oracle for sensitive files. Concretely: with `paths: ["/dir/*"], max_files: 1`, a directory containing only one legitimate `.docx` would give `files_truncated: false`, but adding a `.env-secret.docx` alongside it would push the pre-denylist count to 2 and flip `files_truncated: true` — a directly observable signal that the sensitive file exists. Rev 11 fixes this by counting only post-denylist survivors. The cost is that denylist evaluations now run on every expansion result up to whatever the expansion step produced; the expansion cap at `RELAY_OFFICE_MAX_GLOB_EXPANSION` already bounds that set, so we do not lose termination guarantees.
5. **Apply `reject_sensitive_file_path` to every candidate path (per step 4.1 and 4.3).** `read_file` enforces this at its entry (`apps/desktop/src-tauri/crates/runtime/src/file_ops.rs:153`); `office_search` must replicate that gate explicitly because glob expansion can materialize paths the caller did not name. Rejected paths are dropped from the candidate set silently — **no information about the rejection is returned to the tool caller**. Specifically:
   - The tool result does **not** include per-path rejection entries, a count of rejected paths, a boolean "some-rejected" flag, or any field whose value changes when a glob like `paths: ["/somewhere/.env*"]` matches zero vs. N sensitive files. Returning even a count turns `office_search` into a sensitive-path existence oracle (`.env*` matching → non-zero count → file exists), which is the exact probing behavior we aim to prevent.
   - If the *entire* candidate set was rejected, the tool returns the same empty-result shape it would for "glob matched nothing legitimate," with no distinguishing field. From the tool caller's perspective "you asked about a path that doesn't exist" and "you asked about a path that exists but is sensitive" are indistinguishable.
   - **`files_truncated` is computed on the post-denylist surviving set and therefore safe to surface (rev 11).** Truncation runs at step 4.4, *after* the pre-canon + canonicalize + post-canon denylist checks of steps 4.1–4.3. A sensitive file in the expansion results is silent-dropped before it ever contributes to the count, so toggling a sensitive file's existence cannot flip the flag. What `files_truncated` conveys is "after all silent drops, the remaining legitimate candidate count still exceeded `max_files`" — information the caller could already obtain by counting hits on a legitimate corpus, and therefore not a leak.
   - **Operator visibility requires new plumbing — it is not a pre-existing sink.** The current `reject_sensitive_file_path` implementation (`apps/desktop/src-tauri/crates/runtime/src/tool_hard_denylist.rs:356`) only returns `io::ErrorKind::PermissionDenied`; it does not log, does not rate-limit, and there is no "runtime diagnostics channel that already receives denylist hits" today. Rev 5 referenced such a channel; rev 6 retracts that. Two options:
     - **Option DNS-A (chosen default, implemented in A5):** add a thin private logging sink inside `runtime` — a module-local `once_cell` `Mutex<HashMap<PathHash, Instant>>` rate limiter that emits a single `tracing::warn!(target = "relay::office::diag::denylist", ...)` entry per path per `RELAY_OFFICE_DIAGNOSTICS_LOG_WINDOW_SECS` (default 300). **Gated on `RELAY_OFFICE_DIAGNOSTICS_LOGGING=1` (rev 9 unification).** Rev 5 called this `RELAY_OFFICE_DENYLIST_LOGGING` and the pptx malformed-notes contract at line 116 called its sibling switch `RELAY_OFFICE_DIAGNOSTICS_LOGGING`; rev 9 collapses both into one env var (`RELAY_OFFICE_DIAGNOSTICS_LOGGING`) to avoid the "implementer wires one and forgets the other" footgun the two-name design invited. Individual diagnostics use distinct `tracing` subtargets (`relay::office::diag::denylist`, `relay::office::diag::pptx_notes`, and so on) so operators who want only one stream can filter at the tracing layer rather than at the env layer. Default builds still emit nothing at all. The sink is private to `runtime` — it is not surfaced via IPC, IPC event streams, or any tool result field. This is tracked as a concrete Phase A5 task, not a "use the existing thing" note. **If implementation slips, ship A5 without the log sink rather than retrofitting a leak-prone shortcut** — silent rejection is acceptable; almost-silent rejection is not.
     - **Option DNS-B (fallback):** drop operator visibility entirely for Phase A. Silent is strictly safer than any audit channel we could build carelessly. Pick this if the A5 timeline can't accommodate the sink design-review.
   - Until Option DNS-A lands, `office_search` must behave exactly as if DNS-B were chosen — **no log, no count, no field, no side channel** — and the release notes call out that denylist visibility is a follow-up, not a shipped feature. This prevents the "implementer sees a reference to existing infra that doesn't exist and invents a quick shim that leaks" failure mode.
6. For each surviving candidate, load cache or run `office::extract` (cache-writes on success).
7. Run the regex over each `anchor.text`.
8. Emit hits as:

   ```json
   {"path": "...", "anchor": "Sheet1!B7", "match": "Q3 forecast", "preview": "...Q3 forecast revised down..."}
   ```
9. Total match count capped by `max_results`; report truncation via `results_truncated: true` alongside `files_truncated` from step 4.4. The two truncation flags are independent — a call may hit neither, either, or both.

**Parse-failure surfacing.** Distinct from the silent sensitive-path rejection in step 5, parse failures (corrupt zip, oversized sharedStrings, layer-2 deadline hit, layer-1 limit breach) are reported in a `errors` array on the tool result, one entry per candidate path that failed: `{"path": "...", "kind": "InvalidData" | "TimedOut" | "ResourceBusy", "reason": "<short error text>"}`. This is safe for two reasons:
- These paths survived `reject_sensitive_file_path` in step 5, so they are not by definition sensitive. Returning the path name to the caller does not leak a gate we just closed.
- Parse failures are characterised by a reason string, not by file existence. A "zip central directory invalid" error tells the caller the file exists **and** is malformed; that's information the caller already had to supply the path to obtain, and is useful for actionable error handling ("this .docx is corrupt") without giving the caller a way to probe anything they couldn't already see with `read_file`.

The one error kind that is **not** fanned out per-path is sensitive-path rejection — it stays silent per step 5 and does not appear in the `errors` array. If a future failure mode ever depends on information the caller could not otherwise learn (e.g. some hypothetical "file is sensitive only at this time of day" rule), it must be routed through the same silent channel, not through `errors`. The general rule is: `errors` may name paths the caller already knows about; it must never reveal a path's *existence* or *category* where the caller has no other way to find out.

**Not in this tool**
- Recursive directory walking without an explicit glob (use `paths: ["/dir/**/*.docx"]` style, mirroring `grep_search`)
- Ranking by relevance — results are file-order / anchor-order. LLM iterates.

## Long-running work

Office extraction on first access is synchronous from the *agent's* perspective — `office_search` is a single blocking tool call — but internally the tool drives a small worker pool backed by the same `extract_with_timeout` / per-path reclaim / global-semaphore machinery described under **Stopping, limits, and cancellation**. `office_search` is an **expected-slow tool**, not a sub-second one.

### Tokio runtime contract

`office_search` and the Office-backed `read_file` paths are blocking APIs. They use `std::thread` workers internally, but their public call waits on `std::sync::mpsc::recv_timeout` and can occupy the calling thread for up to `RELAY_OFFICE_SEARCH_MAX_WALL_SECS + RELAY_OFFICE_PARSE_TIMEOUT_SECS` (about 12 minutes with Phase A defaults) for `office_search`, or one `RELAY_OFFICE_PARSE_TIMEOUT_SECS` for a single-file read. Any caller that is already running inside a Tokio task must call these paths through `tokio::task::spawn_blocking` (or the runtime's existing blocking-tool wrapper if one exists by A5), then await the join handle from the async side.

This is a runtime-starvation contract, not an optimization note. It is incorrect to call `tools::execute_tool` for `office_search` directly on a Tokio worker thread if that call can reach the synchronous tool body. The implementation checklist for A5 must verify the tool-dispatch path and add the `spawn_blocking` boundary at the outermost tool-execution layer that still has access to Tokio. Nested `std::thread` workers do not protect Tokio worker threads from the outer synchronous wait.

### How `office_search` drives the worker pool (rev 11 rewrite)

`office::extract_with_timeout(path)` is **synchronous from its caller's perspective** — internally it spawns a `std::thread` worker and blocks on `recv_timeout` until the worker sends a result or the deadline fires (see **Stopping, limits, and cancellation → External wall-clock enforcement**, `apps/desktop/src-tauri/crates/runtime/src/office/mod.rs` once it exists). Rev 10's spec called that function directly from a single tool-side event loop and claimed parallelism would emerge; it wouldn't. The first candidate's call would block the loop, `in_flight` would never exceed 1, and the process-wide concurrency cap would be dead weight.

Rev 11 fixes this by having `office_search` spawn **its own pool of `std::thread` workers**, each of which calls `extract_with_timeout` synchronously. Parallelism comes from the tool-side workers, not from a non-blocking variant of `extract_with_timeout`. Concretely:

1. **Channel setup.** Create two channels: `(work_tx, work_rx)` for dispatching paths to workers, and `(result_tx, result_rx)` for collecting `WorkerOutcome` (either the `ParsedDoc` or an error). **`work_tx` must be a rendezvous channel with capacity exactly 0 (rev 14 tightening).** Rev 13 allowed a carve-out for "tight-bounded capacity up to `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS`," but that re-introduced the exact bug the lazy-enqueue rewrite was meant to kill: a bounded `send` returns successfully as soon as there is space in the buffer, which means the main thread can queue up to `cap` candidates ahead of demand, and on wall-clock breach those already-buffered candidates are still consumed by workers even after the dispatch loop stops calling `send`. Rev 14 retracts the bounded-capacity allowance. Concretely: use `crossbeam_channel::bounded(0)` or `std::sync::mpsc::sync_channel(0)`. **Both `std::sync::mpsc::channel()` (unbounded) and any `bounded(n)` with `n > 0` are rejected at review.** `result_rx` can be unbounded — results arrive only after a worker consumed a send from `work_tx`, so the backlog is already bounded by `outstanding ≤ cap`. Wrap `work_rx` in `Arc<Mutex<_>>` so multiple workers can share it, or use the `crossbeam_channel` multi-consumer form directly — the design-stable choice is deferred to A5 implementation; either is a drop-in.

   **Why capacity 0 and not `cap`.** The rev 13 invariant the dispatch loop is meant to enforce is "the number of candidates that have left the caller's hand but are not yet consumed by a worker is ≤ 0 at every wall-clock check." A rendezvous channel makes this a channel-level property: a successful `send` on capacity 0 is proof that a worker already took the candidate, so there is no buffered-but-unreceived backlog a late cap-breach could race against. A bounded `cap`-capacity channel instead gives a strictly weaker property — "≤ cap pending" — which means cap-breach has to race the workers for the buffer contents, and the "no hidden backlog" wording in **On breach** step 1 becomes untrue in general. The cost of rendezvous is that `work_tx.send` blocks until an idle worker exists; in practice this is exactly what the dispatch loop wanted anyway (it was already maintaining `outstanding < cap` before each send), so the blocking behavior adds no new wait — it just makes the channel refuse to accept candidates the dispatcher had not yet verified it had capacity for.
2. **Pool spawn.** Spawn exactly `min(candidates.len(), RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS)` worker threads via `std::thread::spawn`. Each worker loops:
   1. Receive a `path` from `work_rx`. If the channel closed, exit.
   2. Call `office::extract_with_timeout(path)` **synchronously**. The worker blocks here, which is fine — it is this worker's job to block. During the call, `extract_with_timeout` internally spawns its own reclaim-worker per the layer-3 machinery; those threads do **not** count as `office_search` workers.
   3. Send the `(path, result)` onto `result_tx`. Loop.
3. **Dispatch + multiplexing loop (tool thread; rev 13 lazy-enqueue rewrite).** The main `office_search` thread runs a single loop that maintains the invariant `outstanding ≤ RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS` at every iteration. `outstanding` starts at 0 and counts sends to `work_tx` minus receives from `result_rx`. Let `candidates` be the post-denylist surviving list (already truncated to `max_files` and sorted lexicographically per Behavior step 4.4), iterated by position. On each iteration:

   1. **Wall-clock check.** If `elapsed ≥ RELAY_OFFICE_SEARCH_MAX_WALL_SECS`, set `cap_fired = true` and break out of this loop to the shutdown protocol. Do **not** send another candidate. This is the only place new sends are gated, and because `work_tx` has capacity exactly 0 (rev 14 rendezvous requirement; bounded(n>0) is rejected at review), a successful `send` proves a worker has already taken the candidate — there is no hidden backlog left to drain past the cap.
   2. **Try to enqueue the next candidate, if any, and if there's room.** When `cap_fired` is false, `candidates_remaining > 0`, and `outstanding < RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS`:
      - Call `work_tx.send(next_candidate)`. The rendezvous channel blocks until a worker reads, which can only happen when `outstanding < cap` in practice (an idle worker is parked on `work_rx.recv()`). After the send returns, advance the candidate cursor and set `outstanding += 1`. The rev-13 wording that also described a "small-bounded channel returns immediately because there's space by the invariant" branch is retracted by rev 14 — bounded(n>0) would let the dispatch loop put up to `n` candidates into a buffer ahead of worker demand, which breaks the "no hidden backlog" property the wall-clock cap relies on. There is only one dispatch mode: rendezvous.
      - Do not send more than one candidate per loop iteration — the recv step below must get a chance to observe wall-clock before the next enqueue.
   3. **Receive a result, bounded by the remaining wall-clock budget.** Call `result_rx.recv_timeout(min(remaining_wall_clock, POLL_INTERVAL))`, where `POLL_INTERVAL` is a small constant (e.g. 100 ms) that bounds how long the loop can go without re-checking the cap in the degenerate case where no worker has completed yet. On success, run the regex over the returned anchors, record hits (capped by `max_results`), set `outstanding -= 1`, and loop. On `RecvTimeoutError::Timeout`, loop without blaming a worker — the next iteration will re-check wall-clock and possibly enqueue another candidate if room opened up elsewhere (it won't, but the invariant is the same). On `RecvTimeoutError::Disconnected`, all workers have exited; exit the loop to shutdown.
   4. **Completion detection.** When `candidates_remaining == 0 && outstanding == 0`, the run has finished naturally; break out to the normal shutdown path.

   Because the only way for a candidate to leave the candidates list is a successful `work_tx.send`, and every `send` is gated by the wall-clock check in step 3.1, **the wall-clock cap actually stops new extraction work** — rev 11's "drop `work_tx`" trick, which did not clear an already-queued FIFO, is no longer load-bearing. Dropping `work_tx` in shutdown still matters (it signals workers to exit), but it no longer needs to stop a queue that now does not exist.
4. **Shutdown protocol.** Two scenarios:
   - **Normal (`candidates_remaining == 0 && outstanding == 0`).** Drop `work_tx` so workers observing `work_rx.recv() → Err(Disconnected)` exit. Main thread `join`s each worker handle — since `outstanding == 0`, every worker is either idle waiting on `work_rx` (exits immediately on the drop) or has just finished its last extraction and is about to send on `result_tx`; this wait is bounded by one `result_tx.send` plus the mutex/channel handoff. Results are already collected; emit them.
   - **Cap fired.** Drop `work_tx` immediately so no worker picks up a new path. Each worker is now either idle (exits immediately on `Disconnected`) or mid-extraction (bounded by `RELAY_OFFICE_PARSE_TIMEOUT_SECS` per the layer-3 contract). Main thread `recv_timeout`s on `result_rx` for one additional `RELAY_OFFICE_PARSE_TIMEOUT_SECS` (the drain window documented under **Real upper bound**), folding in any results that arrive and decrementing `outstanding`. Workers that haven't finished by then are abandoned: main thread stops waiting on `result_rx`, emits the synthetic `(wall-clock cap)` error with an accurate `outstanding` count as the "abandoned" number (and `candidates_remaining` as the "still pending" number, since those never entered the pool), and returns. The abandoned workers continue to run until their own `extract_with_timeout` returns (success or per-file timeout), which by the layer-1/2/3 structural limits is bounded. They drop their results silently — nothing is listening on `result_tx` anymore.

   **Why the counts are now accurate.** With rev-11 unbounded `work_tx`, "pending" was ambiguous because it mixed "candidate the tool owned but never gave to a worker" with "candidate a worker had started but not finished." Rev 13 makes this explicit: `candidates_remaining` is the first category (never enqueued), `outstanding` is the second (in a worker). The synthetic error reason uses both numbers verbatim so the caller can distinguish "work we chose not to start" from "work we started but abandoned." This matters because the former is cheap to retry (no semaphore permits held, no reclaim slots occupied), while the latter imposes the permit-hold UX consequence documented below.

This is still single-threaded from the agent's perspective: the tool call is one blocking call, and the worker pool is internal. No task-queue service, no background-job infrastructure, no IPC fan-out. What changed from rev 10 is the acknowledgment that parallelism inside a synchronous tool requires `office_search` to own its own threads; the concurrency cap at `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS` is *also* respected because each worker still goes through `extract_with_timeout`, which still acquires the process-wide semaphore.

**Threading cost.** In the current Phase A shape, one candidate can temporarily occupy two Rust threads: an `office_search` pool worker plus the inner `extract_with_timeout` extraction/reclaim worker. This is an intentional simplicity tradeoff for A5. It keeps the extraction timeout API shared with `read_file` and avoids introducing an async or handle-returning extraction surface before there is evidence that thread count is the bottleneck. If field data shows thread pressure, the follow-up design should evaluate either (a) a handle-returning `extract_with_timeout` variant that lets `office_search` multiplex completion without a pool worker per in-flight extraction, or (b) a direct synchronous extraction path used only by `office_search` workers that relies on the outer worker for timeout ownership.

**Implication for the `extract_with_timeout` API (no change required).** The function keeps its synchronous signature `fn extract_with_timeout(path) -> io::Result<ParsedDoc>`. No non-blocking variant, no handle-returning variant, no Future surface. This matches Phase A's "no new async runtime" posture in the Dependencies section.

### `RELAY_OFFICE_SEARCH_MAX_WALL_SECS` (rev 9 addition; tool-wide cap)

- Default **600s (10 minutes)**, bounds `[10, 3600]`, parsed with the same helper as `RELAY_OFFICE_PARSE_TIMEOUT_SECS`.
- Enforced at the `office_search` tool layer (the event loop above), not inside `office::extract_with_timeout`. The tool checks `elapsed` each time it wakes — before enqueuing the next candidate, and after a worker completion event.
- **On breach:**
  1. **Stop enqueuing new candidates immediately.** The dispatch loop's wall-clock check at step 3.1 of **How `office_search` drives the worker pool** fires before any further `work_tx.send`. Because `work_tx` is a **rendezvous channel with capacity 0** (rev 14 tightening of the rev 13 "rendezvous or tight-bounded" wording), there is **no queued-but-not-yet-consumed backlog** to freeze — every prior successful `send` is proof a worker already consumed that candidate, so "stop enqueuing" is sufficient to stop new work. Rev 11 relied on dropping an unbounded `work_tx` to stop workers from picking up queued candidates, which did not in fact clear the queue; rev 13 fixed the backlog in principle but permitted a bounded-capacity variant that still allowed up to `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS` sends to buffer ahead of demand; rev 14 removes that carve-out so the "no hidden backlog" property holds strictly at the channel level, not just as an invariant maintained by the dispatcher.
  2. **Drain in-flight workers, bounded by one extra `RELAY_OFFICE_PARSE_TIMEOUT_SECS`.** The `outstanding` workers (those that already consumed a `work_tx.send` and are inside `extract_with_timeout`) are allowed to run to natural completion, then their results are folded in. The tool does **not** start a new per-candidate timeout for them — they already have one from `extract_with_timeout`. If an in-flight worker has not produced a result within one additional per-file timeout from the moment the cap fired, the tool abandons it: the detached-worker reclaim machinery at **Cache write safety → Per-path in-flight guard** keeps running in the background until the worker exits, but `office_search` itself moves on.
  3. **Emit results.** Accumulated hits are returned with `wall_clock_truncated: true` alongside `files_truncated` / `results_truncated`. Partial results are useful — matches already found are real.
  4. **Append a synthetic `errors` entry.** `{"path": "(wall-clock cap)", "kind": "TimedOut", "reason": "RELAY_OFFICE_SEARCH_MAX_WALL_SECS=<n>s exceeded; <candidates_remaining> files still pending (never started), <outstanding> abandoned in flight"}`. The synthetic `"path"` is the literal string `(wall-clock cap)` — it is not a real file path and cannot be confused with a per-file parse failure. The two numeric fields are distinct (rev 13): `candidates_remaining` is the count of post-denylist survivors the dispatch loop chose not to enqueue once the cap fired, and `outstanding` is the count of workers that were mid-extraction at cap-fire time, including any that the drain window abandoned. A caller scripting retries uses these separately — pending-never-started work is cheap to re-submit, while abandoned-in-flight work implies the global semaphore may still be holding permits for those paths until their `extract_with_timeout` calls return.

### Real upper bound (rev 10 correction; rev 13 reaffirmed against lazy enqueue)

Honest upper bound per call, accounting for the fact that the wall-clock cap is polled *between* worker events (not inside the kernel of `extract_with_timeout`):

```
wall_clock ≤ min(
    ceil(max_files / RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS) * RELAY_OFFICE_PARSE_TIMEOUT_SECS,
    RELAY_OFFICE_SEARCH_MAX_WALL_SECS + RELAY_OFFICE_PARSE_TIMEOUT_SECS
    // ^^^ the cap + the drain window for the last ≤RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS
    //     in-flight workers that started before the cap fired. With lazy enqueue, no extra
    //     candidate gets started after the cap — the drain window is bounded by the per-file
    //     timeout of workers already inside extract_with_timeout, not by the length of a queue.
)
```

With Phase A defaults (`max_files = 50`, cap = 4, per-file timeout = 120s, tool wall-clock cap = 600s / 10min) the worst case is **roughly 12 minutes** (600s cap + up to 120s drain), not the flat 10 minutes rev 9 claimed. The drain window is necessary because `office_search` has no way to *cancel* an in-flight `std::thread` worker — per the layer-3 contract, Rust threads are not safely interruptible — and forcibly abandoning in-flight results at the cap moment would mean every call that hits the cap discards up to `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS` files of work that were about to succeed. **Rev 13 makes this ceiling load-bearing, not advisory.** Rev 11 computed the same ceiling but the dispatch loop violated it: pre-queuing all candidates onto an unbounded `work_tx` meant workers kept pulling and starting new extractions after the cap fired, so the actual wall-clock could stretch to `ceil(max_files / cap) * per_file_timeout` (up to hours at older defaults) even when the tool-level cap was ostensibly 600s. Rev 13's lazy-enqueue dispatch (see **How `office_search` drives the worker pool** step 3) ensures the cap check runs before every `work_tx.send`, so "no worker picks up a new path after the cap" is actually true at the channel level and the 12-minute ceiling is what callers will observe. Release notes must label this clearly so users do not mistake a long `office_search` call for a hang.

- The cap is orthogonal to `RELAY_OFFICE_PARSE_TIMEOUT_SECS`: a single pathological file can still burn its full 120s inside the 600s budget. Four such files in parallel would consume the full 10-minute budget on their own, at which point the caller sees partial/empty results and a `wall_clock_truncated` flag, and the tool waits up to another 120s for those four workers to finish.
- **Why a tool-level cap on top of the per-file one.** Rev 8 relied solely on `ceil(max_files / cap) * per_file_timeout`, which produced the "minutes, not hours" claim that was numerically wrong at the documented defaults. Rev 9 adds a single wall-clock ceiling so the *tool* has a bounded cost regardless of how `max_files` or the per-file timeout move over time; rev 10 corrects the ceiling to include the drain window.

**Phase A ships without a background-job path; this is deliberate (rev 8 correction, unchanged in rev 9).** Earlier revs claimed `office_search` could be wrapped with the existing `task_registry` / `Task*` tools "with no new infrastructure required." That claim was wrong: `apps/desktop/src-tauri/crates/runtime/src/task_registry.rs:1` is explicitly "in-memory task records (claw-style `Task*` tool surface; no external worker)" and its `TaskStatus` values (`apps/desktop/src-tauri/crates/runtime/src/task_registry.rs:23`) are written by callers, not by a runtime worker. Pointing `office_search` at it would give the agent a task ID that never progresses, no cancellation, and no output — strictly worse than the synchronous path it already has. The correct Phase A posture is: `office_search` is a synchronous tool with a tool-level wall-clock cap; if a caller wants to give up early, they kill the tool call, which is exactly what the existing timeout + concurrency-cap + `ResourceBusy` + `RELAY_OFFICE_SEARCH_MAX_WALL_SECS` machinery already supports.

**If field usage proves the sync path inadequate, Phase B+ adds a real background-job path — not by retrofitting `task_registry` in place.** The required work is non-trivial: a worker thread pool (or `tokio::task::spawn_blocking` queue) that actually executes jobs, a status channel wired into `TaskStatus` transitions, a cancellation token plumbed through `office::extract_with_timeout`'s reclaim machinery, and an output surface that can stream partial results back to the agent. None of that exists today. Treat it as a separate design doc, not a Phase A checklist item. Until then, callers who need "fire and forget" against a huge corpus should script `office_search` over batched `paths` lists on the agent side — the concurrency cap already keeps that from stampeding the runtime.

Assert: (a) the call returns in `3s + drain_window ≤ 123s` (in practice ~4s, since the last in-flight pair finishes within ~1s of the cap fire), **not** `ceil(50/2) * 1s = 25s` — the 25s case is the rev-11 regression where pre-queued candidates kept being consumed past the cap. (b) The synthetic `"(wall-clock cap)"` entry's reason string reports `candidates_remaining > 0` (most of the 50 were never started) and `outstanding ≤ 2` (only the workers mid-extraction at cap-fire were abandoned). (c) `wall_clock_truncated` is true and the hit list contains matches from roughly the first `3 * 2 = 6` files, not from all 50. This fixture is the direct regression for the pre-queued-FIFO bug fixed in rev 13.
   - **Bounded work channel invariant (rev 13 addition).** At any point during the run of the previous fixture, instrument the dispatch loop to assert `outstanding ≤ RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS` and that the number of sends to `work_tx` never exceeds receives from `result_rx` plus `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS`. A failure here indicates the channel was sized too large and the lazy-enqueue invariant is not actually enforced. This is a white-box test that can live in a unit test against the internal dispatch helper if full black-box instrumentation is awkward.
   - **PDF header does not leak into cache anchor (rev 9 addition):** index a `.pdf` through `office_search`, then regex for a substring known to be in the LiteParse banner (e.g. `"OCR disabled"`). Assert zero matches from the PDF (the header is not in the cache) — regression for the rev-7 bug where `read_pdf_as_text` was referenced and would have embedded the banner.
6. **Phase A6** — **cross-format `compat-harness` end-to-end coverage**: multi-file corpora mixing docx/xlsx/pptx/pdf, agent-level workflows that exercise `office_search` + `read_file` round-trips, combined-stress fixtures (e.g. a directory with a zip bomb next to a valid docx, asserting the valid one still resolves). The per-format adversarial fixtures already shipped in A1–A3 and A5 above; A6 does not re-home them.
7. **Phase A7 (follow-up, optional)** — page-aware PDF anchors: extend `parse.mjs` to emit `{page, text}` JSONL, add `pdf_liteparse::read_pdf_as_pages()`, switch `office::extract`'s `.pdf` arm to produce one anchor per page. Removes the PDF/Office citation asymmetry from Phase A.

Each sub-phase ships independently; `read_file` integration is additive and does not break existing `.pdf` behavior.

## Release notes requirements

Phase A release notes must include these operational caveats:

- `office_search` is a blocking, expected-slow tool. With defaults, a worst-case call can take roughly 12 minutes (`RELAY_OFFICE_SEARCH_MAX_WALL_SECS=600s` plus one `RELAY_OFFICE_PARSE_TIMEOUT_SECS=120s` drain window).
- After a timeout or wall-clock cap, background reclaim workers can keep semaphore permits until they exit. Follow-up Office reads/searches may return transient `ResourceBusy` for several minutes; callers should retry later rather than treating it as data loss.
- XLSX formatting is best-effort. Search/read output prioritizes cell text recall over exact Excel display formatting for currencies, dates, and locale-specific number formats.
- PDF search anchors are document-level (`doc`) in Phase A. Per-page PDF anchors are deferred to Phase A7.

## Open questions (defer until Phase A1 implementation starts)

Release notes must call out that currency / date formatting may differ from Excel's display. This is the same decision reflected in the xlsx extraction bullet and the xlsx `read_file` header above — they were reconciled in rev 5 after an earlier draft overclaimed "formatted cell values" in the header while the open question still defaulted to raw. Revisit Option Y post-A2 only if recall on finance-heavy workbooks proves inadequate.

Per-row keeps anchor counts sane; revisit if search recall suffers. **Sub-question resolved in rev 14 (anchor scope):** body paragraph anchors are top-level only (parent is `<w:body>` or an `<w:sdt>` wrapper of `<w:body>`); table cell paragraphs are owned by the row anchor exclusively. See **Extraction details → docx** for the exact rule. The per-row-vs-per-cell tradeoff is still open.
3. **PPTX slide anchors.** Whether to include master/layout text. Default: skip (they're template boilerplate).
4. **Cache eviction.** Whether Phase A needs any size budget. Default: no, revisit on field reports.
5. **Path canonicalization under symlinks.** `canonicalize` resolves symlinks, which is what we want for cache keying but surprises users who expect "same link, same cache." Document current behavior; revisit if it causes support churn.
6. **Detached-worker safety on Windows.** The "detach the thread on timeout" pattern relies on bounded structural limits to ensure the worker eventually exits. Validate under Windows with bundled `relay-node`-adjacent tooling to confirm no file-lock fallout (Office files held open elsewhere).
7. **Concurrency-cap tuning.** The default `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS = min(4, cpus-1)` was picked conservatively in rev 5. Revisit once we have field data on typical corpus walks — if most callers never hit the cap, raise it; if `ResourceBusy` is a frequent user-visible error, either raise the cap or add a brief in-process wait (bounded by `RELAY_OFFICE_PARSE_TIMEOUT_SECS / 4`) before returning `ResourceBusy`. Resolved in rev 5 for the "do we need one at all" question; tuning is the open part.
8. **Per-file parse-failure surfacing — resolved in rev 5.** `office_search` returns per-path `errors` entries for candidates that survived `reject_sensitive_file_path` but failed to parse or timed out. Sensitive-path rejections remain silent (step 2 of the Behavior list). The rule for future failure kinds: they join `errors` only if the caller already had a way to learn the path exists. Listed here so the design rationale is visible in one place; not an open question.
9. **`include_ext` pipeline position — resolved in rev 7, refined in rev 8.** Validate `include_ext` (lowercase, strip leading `.`, reject entries outside `{docx, xlsx, pptx, pdf}` with `InvalidInput`) at input-validation time. Extension filtering applies at two places: (a) pattern-level, for *concrete* `paths` entries that can be rejected before any filesystem call (Behavior step 2), and (b) during glob traversal as a cost-minimization step (Behavior step 3). Rev 8 retracts the earlier claim that extension filtering runs "before any I/O or security check" for all inputs — that holds only for concrete paths. For glob inputs, glob expansion is itself filesystem work, and the authoritative security boundary is the per-candidate denylist at Behavior step 4.2, not the extension filter. Rationale: treating the extension filter as load-bearing for security would invite implementers to skip the denylist for "already-filtered" extensions.
10. **`office_search` wall-clock budget — resolved in rev 8, defaults corrected in rev 9, stale numbers removed in rev 10.** `max_files` (input; **default 50, hard cap 1_000** per the Inputs section; rev 10 removed the stale `default 500, hard cap 5_000` that this entry used to repeat) plus `RELAY_OFFICE_MAX_GLOB_EXPANSION` (default 50_000, hard cap 250_000) bound the total extraction work per call. Combined with `RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS`, `RELAY_OFFICE_PARSE_TIMEOUT_SECS`, and `RELAY_OFFICE_SEARCH_MAX_WALL_SECS`, wall-clock is bounded; see **Long-running work** for the full arithmetic. Revisit the defaults once field data shows where typical corpus walks sit.
11. **PPTX malformed-notes contract — resolved in rev 8.** Malformed `notesSlide{N}.xml` omits the `slide{N}:notes` anchor silently; the body anchor still extracts. `ParsedDoc` stays anchor-only. Whole-file body-parse failures still fail the whole file with `InvalidData`. See the pptx extraction section for the full contract.
12. **Background-job path for `office_search` — resolved in rev 8 (as a non-goal for Phase A).** `task_registry` cannot back long-running extraction; rev 7 incorrectly claimed it could. Phase A ships synchronous-only with a bounded wall-clock budget (see items 10 and **Long-running work**). A real background-job path is Phase B+ design work, not a Phase A checklist item.
13. **Concurrency-cap acquire is fail-fast, not queued — resolved in rev 8, rev 9 cleaned up stale "blocking-park" text.** Permit miss returns `ResourceBusy` immediately with no internal wait; the earlier "FIFO fairness" and "try_acquire / blocking-park" phrasings are both retracted because there is no queue and no park. Callers treat `ResourceBusy` as a normal outcome under contention. See **Process-wide extraction concurrency cap → Acquire semantics**.
14. **Sensitive-path denylist scope — basename-only today (rev 9 explicit gap, rev 10 named anchor).** `reject_sensitive_file_path` at `apps/desktop/src-tauri/crates/runtime/src/tool_hard_denylist.rs:356` matches on basename (`.env*`, `id_rsa*`) and extension (`.key`, `.pem`) only. A `.pdf` sitting inside `~/.ssh/` is **not** blocked by the current implementation, because its basename does not match any pattern. Earlier design drafts assumed `.ssh` / `.gnupg` directory-prefix skipping existed; that was incorrect. Phase A `office_search` aligns with the actual basename-only behavior. Extending the denylist to cover directory-prefix patterns is a separate, larger change (it affects `read_file` / `write_file` / `edit_file` too, not just `office_search`) and is deferred. If Phase A5 review decides we want directory-prefix coverage for the new tool surface, the right answer is to extend the shared denylist, not to invent an office-local parallel rule set. Track alongside the Option DNS-A diagnostics-sink work. **(Cross-referenced as "Sensitive-path denylist scope" from the glob-expansion section; cross-links point at the named anchor, not the numeric position, so the list can be renumbered without breaking references.)**
15. **PDF dispatch uses the payload-only API — resolved in rev 12.** `office::extract` for `.pdf` calls `read_pdf_as_payload`; it does **not** call `read_pdf_as_text`, because that function prepends the LiteParse banner header and we do not want the header text showing up as cacheable anchor content every PDF could match against. `read_file` for `.pdf` keeps calling `read_pdf_as_text` to preserve today's user-visible output. Phase A4 adds `PDF_LITEPARSE_HEADER` plus `read_pdf_as_payload` and rewrites `read_pdf_as_text` over that payload helper.
16. **Sensitive-literal globs silent-dropped before expansion — resolved in rev 9.** The glob pattern's literal prefix is denylisted before `walkdir` / `glob` touches disk; a hit silent-drops the entry rather than reaching the expansion-cap `InvalidInput` path that would be an existence oracle. See Behavior step 3 under `office_search`.
17. **Tool wall-clock cap (`RELAY_OFFICE_SEARCH_MAX_WALL_SECS`) — resolved in rev 9, real upper bound clarified in rev 10.** Rev 8's wall-clock arithmetic was numerically off by roughly an order of magnitude (claimed "minutes", defaults produced hours). Rev 9 drops `max_files` default to 50 / hard cap to 1_000 and adds a 10-minute tool-level cap with `wall_clock_truncated` partial-results reporting. Rev 10 clarifies that the real per-call upper bound is `RELAY_OFFICE_SEARCH_MAX_WALL_SECS + RELAY_OFFICE_PARSE_TIMEOUT_SECS` (the cap is polled between extractions, so the in-flight worker at cap-breach time keeps running until its per-file timeout). See **Long-running work** for the full contract including how `office_search` drives the worker pool and what "partial return" means.
18. **Diagnostics env var is unified — resolved in rev 9.** A single `RELAY_OFFICE_DIAGNOSTICS_LOGGING=1` switch gates all Phase A diagnostic sinks (denylist hits, pptx malformed-notes omissions, future office-side diagnostics). Per-source filtering uses distinct tracing subtargets under `relay::office::diag::*`. Earlier drafts had two env vars (`RELAY_OFFICE_DENYLIST_LOGGING` for the denylist sink, `RELAY_OFFICE_DIAGNOSTICS_LOGGING` for the pptx notes sink) which invited the "implementer wires one, misses the other" failure mode.
19. **`files_truncated` is computed post-denylist — resolved in rev 11.** Rev 10 counted candidates for the truncation flag before applying `reject_sensitive_file_path`, which made the flag flip on sensitive-file existence — a 1-bit oracle against exactly the contract silent-drop was designed to uphold. Rev 11 moves truncation to Behavior step 4.4, after the per-candidate denylist at 4.1–4.3. The flag now conveys "legitimate-file count exceeds `max_files`" only. See Behavior step 4 and the rev-11 A5 fixture.
20. **Worker pool is owned by `office_search`, not implicit in `extract_with_timeout` — resolved in rev 11.** `extract_with_timeout(path)` is synchronous per design (internally `std::thread::spawn` + `recv_timeout`), so parallelism cannot emerge from calling it in a single event loop. Rev 11 spells out that `office_search` spawns its own pool of `std::thread` workers, each of which calls `extract_with_timeout` synchronously; the process-wide semaphore is acquired inside each worker's `extract_with_timeout` call, so the concurrency cap still applies. See **Long-running work → How `office_search` drives the worker pool**.
21. **Pre-expansion glob denylist is Windows-aware and case-insensitive — resolved in rev 11.** The procedure splits `paths` entries on both `/` and `\` before taking the filename glob, and lowercases the filename glob before comparing against the denylist rules. Rev 10 spec'd `/`-only split and case-sensitive match, both of which would have reopened the side-channel oracle. See Behavior step 3.
22. **Regex compilation happens at input-validation time — resolved in rev 11.** Invalid patterns fail as `InvalidInput` before glob expansion, canonicalization, or extraction. Rev 10 left compilation implicit at search time, paying full pipeline cost on a malformed pattern. See Behavior step 1.
23. **Windows canonical path verbatim prefixes — resolved in rev 12.** Cache keys hash `normalize_cache_path(canonicalize(path))`, stripping Windows `\\?\` / `\\?\UNC\` prefixes before hashing or persisting `source`. This keeps conventional and verbatim spellings of the same canonical Windows path from producing different cache records.
24. **Zip entry-name validation — resolved in rev 12.** Office part lookup is exact, case-sensitive, and only after rejecting non-UTF-8, NUL-containing, absolute, drive-prefixed, backslash-separated, `.` / `..`, and empty-segment entry names. Format-boundary `catch_unwind` converts parser/library panics to `InvalidData`.
25. **Tokio blocking boundary for `office_search` — resolved in rev 12.** The synchronous tool body must be reached through `tokio::task::spawn_blocking` or an equivalent blocking-tool pool when called from Tokio. Internal `std::thread` workers do not make it safe to block a Tokio worker on `recv_timeout`.
26. **Double-thread cost in `office_search` — accepted in rev 12.** A5 keeps the simple shape where an `office_search` pool worker calls the synchronous `extract_with_timeout`, which itself owns an extraction/reclaim worker. This can mean two Rust threads per in-flight candidate, but preserves one extraction API for both `read_file` and `office_search`. Optimize only if thread pressure shows up in field data.
27. **`office_search` dispatch uses lazy enqueue, not pre-queued FIFO — resolved in rev 13, channel tightened in rev 14.** `work_tx` is a **rendezvous channel with capacity exactly 0** (rev 14 retracted the rev-13 "rendezvous or tight-bounded up to `cap`" allowance because a bounded send returns before a worker consumes, which reopens the backlog), and the dispatch loop sends at most one candidate per iteration while maintaining `outstanding ≤ RELAY_OFFICE_MAX_CONCURRENT_EXTRACTIONS`. This makes the wall-clock cap actually stop new extraction work; rev 11's "pre-queue everything and drop `work_tx` on breach" pattern did not stop already-queued candidates from being consumed, so the 12-minute upper bound was advisory only. Rev 13 made it load-bearing in principle; rev 14 closes the bounded-capacity gap so the guarantee holds at the channel level, not just as a dispatcher invariant. See **How `office_search` drives the worker pool** step 3 and the **Wall-clock cap actually stops new work** A5 fixture.
28. **xlsx preflight and `calamine` share a single file handle — superseded by item 31 (rev 15).** This item is kept as historical context only. Rev 13 introduced a shared-`File` scheme (preflight opens → rewinds → hands the same handle to `calamine`) to close the rev-12 path-replace TOCTOU, and rev 14 added `OpenOptionsExt::share_mode(FILE_SHARE_READ | FILE_SHARE_DELETE)` on Windows. Rev 15 retracts that scheme entirely because it cannot defend against same-inode mutation on POSIX. **The current source of truth is item 31 below (bounded in-memory snapshot); do not implement the shared-`File` scheme described here, and do not cite this item as resolution.** Passing a path string to `calamine` remains a review-rejection criterion; passing a `File` handle to `calamine` is now a review-rejection criterion too. See **xlsx preflight** and item 31.
29. **`MAX_OFFICE_TEXT_BYTES` overflow is `InvalidData`, not silent truncation — resolved in rev 13.** Total anchor text exceeding 16 MiB fails the whole extraction with `io::ErrorKind::InvalidData`; no cache record is written and no partial `ParsedDoc` is returned. This matches `pdf_liteparse`'s error-on-overflow convention and prevents the earlier silent short-circuit from turning cutoff content into invisible `office_search` misses. If the 16 MiB cap proves tight for legitimate inputs, raise the cap; do not reintroduce silent truncation. See the Output-size guard bullet under **Stopping, limits, and cancellation → Bounded per-format structural limits** and the sharpened A1 fixture.
30. **DOCX table-aware traversal — resolved in rev 14; row-anchor uniqueness sharpened in rev 15.** Body `<w:p>` iteration is scope-limited: a paragraph anchor `p{n}` is emitted only for `<w:p>` whose immediate structural parent is `<w:body>` (optionally through `<w:sdt>`/`<w:sdtContent>` wrappers). Paragraphs nested under `<w:tbl>` / `<w:tr>` / `<w:tc>` are owned exclusively by the row traversal and emitted as `p{n}:tbl{j}:row{i}` anchors (rev-15 grammar: `j` is the 1-based table index within the part, making row anchors globally unique within the document; rev 14's `p{n}:row{i}` scheme produced byte-identical citations for sibling tables under the same preceding paragraph). This prevents the same cell text from appearing in both a body paragraph anchor and a row anchor, which a naive descendant-walk implementation would have produced, and ensures every row anchor the `office_search` result exposes is round-trippable to a unique row. See **Extraction details → docx** for the full rule and the **DOCX table traversal does not double-count** A1 fixture.

31. **XLSX preflight uses a bounded in-memory snapshot — resolved in rev 15.** Rev 13/14 had preflight and calamine share an open `File` on POSIX, which left a same-inode mutation gap: a concurrent writer with write access to the same inode could rewrite the bytes between preflight-completion and calamine's reads, potentially producing a valid workbook whose sharedStrings.xml or sheet XML exceeds the preflight's per-part byte caps in bytes the preflight never saw. Rev 15 replaces the shared-`File` scheme with a bounded `Vec<u8>` snapshot captured during preflight (hard cap `RELAY_OFFICE_XLSX_ARCHIVE_MAX_BYTES`, default 256 MiB, enforced via `take(cap + 1).read_to_end(...)`), and feeds both the zip container walk and `calamine::open_workbook_from_rs` a `Cursor<_>` over that immutable buffer. On Windows the snapshot read still uses the restricted `share_mode(FILE_SHARE_READ | FILE_SHARE_DELETE)` so a concurrent writer cannot mutate bytes during the read window itself. See **xlsx preflight** and the **Snapshot rejects same-inode mutation** A2 fixture below.

32. **Cache key uses OS-native path bytes, not `to_string_lossy()` — resolved in rev 15.** Rev 12's wording (`sha256(normalize_cache_path(...).to_string_lossy())`) had two failure modes. On POSIX, two distinct non-UTF-8 paths could collide on their `U+FFFD` replacement-character lossy forms and share a single cache record, which a call to `office_search` over a glob like `paths: ["/data/**"]` is plausibly capable of producing. On Windows, `to_string_lossy()` over a WTF-16 path with unpaired surrogates has the same collapse. Rev 15 hashes and persists `source` as OS-native bytes: `as_os_str().as_bytes()` on POSIX, `encode_wide` → little-endian `u16` bytes on Windows. Non-UTF-8 candidates are not separately rejected — they flow through cache lookup and `office::extract` unchanged — but the `office_search` result JSON's `path` field still falls back to `to_string_lossy()` for display, with the understanding that caller-side shell round-trip of such paths is a Phase A limitation. See **Cache → Key design → Path-to-bytes rule**.

## Future: Phase B (semantic)

If retrieval quality turns out to be a real blocker — concept queries, heavy paraphrase, cross-lingual — add an embeddings layer on top of the same `office::extract` / cache pipeline:

- Chunking over `AnchoredText` (merge small anchors, split large ones; target ~500–1000 tokens, 100 overlap)
- Local embeddings first (`fastembed-rs` with a multilingual-e5 variant for JA/EN coverage)
- `sqlite-vec` as the vector store, co-located with the extraction cache
- New tool `office_semantic_search`; `office_search` stays as the exact-match workhorse

Phase B does not change the extraction or cache layer; it layers on top. Nothing in Phase A precludes it.
