■ 現状の問題

  Relay Agent は以下を全て自前で実装している:
  - 21個のビルトインツール (tool_registry.rs)
  - セッション/ターン管理 (storage.rs: 328KB の巨大モノリス)
  - コンテキスト管理 (relay.rs: RelayPacket 生成)
  - ファイル操作 (file_ops.rs: 22KB)
  - ワークブック処理 (workbook/: 5モジュール)
  - MCP クライアント (mcp_client.rs)
  - パーミッション管理 (risk_evaluator.rs)

  これらの多くは claw-code が既に高品質で実装済み。車輪の再発明をやめる。

■ claw-code とは

  GitHub: https://github.com/ultraworkers/claw-code.git
  (実体: claw-cli/claw-code-rust)

  Claude Code の設計パターンを Rust でクリーンルーム再実装した
  オープンソースの AI コーディングエージェントフレームワーク。

  8 crate の Cargo ワークスペース:
  ┌──────────────────┬────────────────────────────────────┐
  │ claw-cli         │ バイナリエントリ、REPL/単発モード     │
  │ claw-core        │ メッセージモデル、セッション、エージェントループ │
  │ claw-tools       │ Tool trait + 6ビルトイン (bash, file_read/write/edit, glob, grep) │
  │ claw-provider    │ ModelProvider trait + Anthropic/OpenAI実装 │
  │ claw-permissions │ PermissionPolicy trait, ルールベース   │
  │ claw-compact     │ トークン予算管理、コンテキスト圧縮      │
  │ claw-tasks       │ バックグラウンドタスク (scaffold)       │
  │ claw-mcp         │ MCP クライアント (scaffold)           │
  └──────────────────┴────────────────────────────────────┘

■ 方針: 薄いラッパー

  claw-code の crate を Cargo 依存として取り込み、
  Relay Agent 固有のロジックだけを薄く上に被せる。

  具体的には:
  1. claw-tools → ファイル操作・bash・glob・grep をそのまま使う
  2. claw-core → エージェントループ・セッション管理を使う
  3. claw-compact → コンテキスト圧縮を使う
  4. claw-permissions → パーミッション管理を使う
  5. claw-provider → 使わない (Claude API 不可のため)
     → 代わりに CopilotChatProvider を自前実装

■ 制約: Claude API が使えない

  claw-code は Anthropic API / OpenAI API を前提としているが、
  この環境では Claude API キーがない。

  解決策: claw-provider の ModelProvider trait を実装した
  CopilotChatProvider を作成。内部で既存の browser_automation.rs
  (Edge + Playwright 経由の M365 Copilot Chat) を呼び出す。

  ModelProvider trait:
    async fn complete(&self, messages, tools) -> Result<Response>
    async fn stream(&self, messages, tools) -> Result<Stream<Event>>

  CopilotChatProvider は complete() のみ実装し、
  stream() は complete() の結果を1チャンクで返す擬似ストリーム。

■ 概要

  claw-provider の ModelProvider trait を実装する。
  内部で browser_automation.rs を使い、M365 Copilot Chat に
  プロンプトを送信し、レスポンスを claw-core の形式に変換する。

■ ModelProvider trait (claw-provider が定義)

  pub trait ModelProvider: Send + Sync {
      async fn complete(
          &self,
          messages: &[Message],
          tools: &[ToolDefinition],
          config: &CompletionConfig,
      ) -> Result<CompletionResponse>;

      async fn stream(
          &self,
          messages: &[Message],
          tools: &[ToolDefinition],
          config: &CompletionConfig,
      ) -> Result<Pin<Box<dyn Stream<Item = StreamEvent>>>>;
  }

■ CopilotChatProvider 実装

  pub struct CopilotChatProvider {
      browser: BrowserAutomation,  // 既存の browser_automation.rs
  }

  impl ModelProvider for CopilotChatProvider {
      async fn complete(&self, messages, tools, config) -> Result<CompletionResponse> {
          // 1. messages + tools を Copilot 向けプロンプトに変換
          //    - システムプロンプト: ツール定義を JSON Schema で埋め込み
          //    - ユーザーメッセージ: 直近のコンテキストを連結
          //    - レスポンス形式: JSON で tool_use を返すよう指示
          let prompt = self.format_for_copilot(messages, tools)?;

          // 2. Copilot Chat に送信
          let raw_response = self.browser.send_to_copilot(&prompt).await?;

          // 3. Copilot のレスポンスを CompletionResponse に変換
          //    - テキスト部分 → TextContent
          //    - JSON tool_use ブロック → ToolUseContent
          self.parse_copilot_response(&raw_response)
      }

      async fn stream(&self, messages, tools, config) -> Result<Stream> {
          // Copilot Chat はストリーミング不可なので
          // complete() の結果を1チャンクで返す
          let response = self.complete(messages, tools, config).await?;
          Ok(once(StreamEvent::Complete(response)))
      }
  }

■ プロンプト変換の要点

  Copilot Chat は Claude API とは異なり、
  tool_use プロトコルをネイティブサポートしていない。

  そのため、プロンプト内で以下を明示:
  1. 利用可能なツールを JSON Schema で記述
  2. ツール呼び出し時は特定の JSON 形式で返すよう指示
  3. レスポンスから JSON ブロックを抽出して ToolUseContent に変換

  既存の agent-loop-prompts.ts のロジックを Rust に移植。

  問題:
  - TS ↔ Rust 間の IPC が往復するたびにオーバーヘッド
  - ループロジックが TypeScript に分散 (3ファイル、~30KB)
  - コンテキスト管理が手動 (RelayPacket で毎回組み立て)

■ After (Rust 側でループ)

  claw-core の query() 関数がエージェントループを担当:
  1. メッセージ履歴 + ツール定義を CopilotChatProvider に送信
  2. レスポンスから tool_use を抽出
  3. ToolRegistry (claw-tools) でツール実行
  4. 結果をメッセージ履歴に追加
  5. コンテキスト超過時は claw-compact で圧縮
  6. end_turn or エラーまで繰り返し

  フロントエンドは Tauri イベントを listen するだけ:
  - "agent:tool_start" → UnifiedFeed にエントリ追加
  - "agent:tool_result" → エントリ更新
  - "agent:approval_needed" → ApprovalCard 表示
  - "agent:turn_complete" → CompletionCard 表示
  - "agent:error" → エラー表示

9. 移行戦略 — 段階的置換
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  一気に書き換えるのではなく、段階的に claw-code に移行する。
  各フェーズで動作するアプリを維持する。

  Phase 1: 依存追加 & CopilotChatProvider
    - claw crates を Cargo.toml に追加
    - CopilotChatProvider (ModelProvider trait) を実装
    - 既存の browser_automation.rs を内部利用
    - テスト: Copilot Chat 経由で claw-core の query() が動くことを確認

  Phase 2: ツール移行
    - claw-tools のビルトイン (bash, file_*, glob, grep) を有効化
    - Relay 固有ツール (workbook.*, document.*) を Tool trait で登録
    - 旧 tool_registry.rs, file_ops.rs を削除
    - テスト: 全ツールが claw-tools 経由で動くことを確認

  Phase 3: エージェントループ移行
    - TypeScript の agent-loop を Rust の claw-core ループに置換
    - Tauri イベント emit でフロントエンドに通知
    - 旧 agent-loop.ts, agent-loop-core.ts, agent-loop-prompts.ts を削除
    - IPC コマンドを大幅削減 (94 → ~20)
    - テスト: E2E でタスク実行が動くことを確認

  Phase 4: ストレージ簡素化
    - storage.rs の大部分を claw-core のセッション管理に移行
    - ワークブック状態管理のみ残す
    - 旧 relay.rs, risk_evaluator.rs を削除
    - テスト: セッション管理が正常に動くことを確認

  Phase 5: マニュアルモード廃止
    - +page.svelte からガイドワークフロー UI を削除 (~857 行)
    - guidedStage, uiMode, 関連ヘルパー関数を削除
    - continuity.ts から UiMode/loadUiMode/saveUiMode を削除
    - テスト: 委任モードのみで全機能が動くことを確認

  Phase 6: クリーンアップ
    - 未使用コード・依存の削除
    - IPC 型定義 (contracts/) の簡素化
    - ドキュメント更新

10. リスクと対策
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ■ リスク 1: claw-code のリポジトリが消える/非互換変更
    対策: fork を作成して固定コミットで依存する

  ■ リスク 2: Copilot Chat の応答形式が不安定
    対策: パース失敗時のリトライ + 修復プロンプト (既存ロジック流用)

  ■ リスク 3: claw-core の trait が Relay の要件に合わない
    対策: 必要なら trait を newtype パターンでラップ

  ■ リスク 4: ストリーミング非対応による UX 劣化
    対策: ツール実行中はフロントエンドにリアルタイムイベントを送る
          Copilot 応答待ち中はスピナー表示

  ■ リスク 5: claw-code の依存 (async-anthropic 等) が重い
    対策: feature flag で Anthropic/OpenAI provider を無効化
          CopilotChatProvider のみコンパイル

12.5 Step Cluster フィード — UnifiedFeed 刷新
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  openwork の最大の特徴: ツール呼び出しをグループ化してコラプシブルに。
  旧 UnifiedFeed の「1行1エントリ」から変更。

  ■ Step Cluster 構造

    エージェントループの1ラウンドを1つの「クラスター」としてまとめる。

    [クラスター 1] ファイル読込  (3ステップ)  1.2s  ▶ (折りたたみ)
    ─────────────────────────────────────────────────────
    [クラスター 2] データ処理    (2ステップ)  0.8s  ▶
    ─────────────────────────────────────────────────────
    [クラスター 3] 保存準備中...              ▼ (展開中)
      ┌──────────────────────────────────────┐
      │  ✓  workbook.inspect   report.csv  0.3s │
      │  ✓  sheet.preview      32行        0.1s │
      │  ⟳  table.filter_rows  実行中...        │
      │  ○  workbook.save_copy 待機中            │
      └──────────────────────────────────────┘

  ■ クラスターヘッダー

    左:  ラウンド番号 + ラベル (最初のツール名から自動生成)
    中:  ステップ数バッジ (rounded-full, gray-100)
    右:  経過時間 (--sz-xs, --c-text-3) + 展開アイコン (▶/▼)
    高さ: 36px (行)
    背景: 通常は透明。ホバーで --c-surface。実行中は accent-subtle の左ボーダー 2px。

  ■ ステップ行 (展開時)

    インデント: 16px
    左: ステータスアイコン
      ✓ (green)  — 完了
      ⟳ (accent) — 実行中 (回転アニメーション)
      ○ (gray)   — 待機
      ✗ (red)    — エラー
    中: ツール名 (mono, --sz-sm) + 引数サマリ (--c-text-2)
    右: 所要時間 (--sz-xs, --c-text-3)
    詳細: クリックでアコーディオン展開 → JSON 表示 (gray-50 背景)

  ■ セクションラベル (openwork 特徴)

    クラスター一覧の上部に uppercase ラベル:

    EXECUTION LOG                    ← 11px uppercase tracked
      [クラスター 1] ...
      [クラスター 2] ...

  ■ UnifiedFeed コンポーネント仕様

    Props:
      clusters: StepCluster[]
      isRunning: boolean

    StepCluster:
      id: string
      roundIndex: number
      label: string          (自動生成: 最初のツール名から)
      steps: StepEntry[]
      status: "running" | "done" | "error"
      startTime: number
      endTime?: number
      expanded: boolean      (ローカル状態)

    StepEntry:
      toolName: string
      argsSummary: string
      status: "pending" | "running" | "done" | "error"
      durationMs?: number
      detail?: string        (JSON レスポンス)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

12.9 Inbox パネル — ファイル共有エリア
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  openwork の "Inbox" パターンを採用。
  エージェントと共有するファイルを一元管理するエリア。
  右コンテキストパネルの FILES セクションを拡張した独立パネル。

  ■ 概要

    Inbox はユーザーがエージェントに「渡す」ファイルの置き場所。
    - ドラッグ & ドロップでファイルを追加
    - セッションをまたいで参照可能
    - エージェントは Inbox ファイルを優先的に参照する

  ■ UI 構成

    右パネルの FILES セクションが Inbox を兼ねる:

    ┌──────────────────────────────┐
    │  FILES                       │  ← uppercase 11px
    │                              │
    │  ┌──────────────────────┐   │
    │  │  ここにファイルをドロップ  │   │  ← ドロップゾーン
    │  │  または [ファイルを選択]  │   │    (点線ボーダー, --r-lg)
    │  └──────────────────────┘   │
    │                              │
    │  📄 report.csv          ×   │  ← 追加済みファイル
    │     32KB · 2分前             │    (サイズ + 追加時刻)
    │  📊 sales_q1.xlsx       ×   │
    │     156KB · 昨日             │
    │                              │
    │  [+ ファイルを追加]           │
    └──────────────────────────────┘

  ■ ファイル行のスタイル

    height: 40px
    padding: 6px 8px
    border-radius: var(--r-md)  /* 16px */
    display: flex, align-items: center, gap: 8px

    左: ファイルタイプアイコン (16px)
    中: ファイル名 (--sz-sm, font-weight 500) +
        サイズ・時刻 (--sz-xs, --c-text-3, 改行)
    右: × ボタン (ghost, 16px, ホバーで出現)

  ■ ドロップゾーン

    通常:
      border: 1.5px dashed var(--c-border-strong)
      border-radius: var(--r-lg)
      background: transparent
      padding: 16px
      text-align: center
      color: var(--c-text-3)
      font-size: var(--sz-sm)

    ドラッグオーバー時:
      border-color: var(--c-accent)
      background: var(--c-accent-subtle)
      color: var(--c-accent)

  ■ エージェントとの連携

    Inbox に追加されたファイルは:
    1. Tauri IPC で session_store に記録
    2. claw-core のシステムプロンプトに自動追記
       "以下のファイルが共有されています: [パス一覧]"
    3. workbook.inspect が初回自動実行される (オプション)

  ■ コンポーネント

    InboxPanel.svelte (右パネル内のサブコンポーネント)
      Props:
        files: InboxFile[]
      Events:
        add: { paths: string[] }
        remove: { path: string }

    InboxFile:
      path: string
      name: string
      sizeBytes: number
      addedAt: number

12.11 インタラクションパターン (更新)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ■ 起動フロー

    1. 3ペインレイアウトで起動
    2. 中央: 空状態 (サジェストチップ + 最近ファイル)
    3. 右パネル: ファイルなし、MCP 接続状態表示
    4. 左: セッション履歴 (初回は空 → 「最初のタスクを始めましょう」)

  ■ タスク実行フロー

    1. コンポーザーにファイルドロップ or テキスト入力 → Enter
    2. 右パネルのファイルセクションにチップが追加される
    3. プラン確認が必要なら ApprovalCard がフィード内に出現
    4. 承認 → Step Cluster が順次表示される
    5. 書込操作に到達 → ApprovalCard (3択) が出現
    6. 「今回のみ許可」選択 → 実行継続
    7. 完了 → CompletionCard

  ■ キーボードショートカット (更新)

    ⌘K:     コマンドパレット (Pipeline / Batch / Template / 設定)
    ⌘\:     左サイドバー表示切替
    ⌘/:     右コンテキストパネル表示切替  ★追加
    ⌘N:     新規タスク (コンポーザーをクリア)
    Enter:   送信
    Shift+Enter: 改行
    Escape:  パレット/承認カードをキャンセル

  ■ コンパクション状態の表示

    claw-compact によるコンテキスト圧縮が発生した際:
    フィード内に薄いインジケーター表示:
    ─── コンテキストを圧縮しました (古い履歴を要約) ───
